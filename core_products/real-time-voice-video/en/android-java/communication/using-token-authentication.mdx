# Using Token Authentication

---

## Feature Introduction

Authentication refers to verifying whether users have permission to access the system, to avoid security risk issues caused by lack of permission control or improper operation. ZEGO authenticates users through Token (including basic authentication Token and privilege authentication Token).


|Authentication Method|Description|Application Scenarios|
|-|-|-|
|[Basic Authentication Token](/real-time-video-android-java/communication/using-token-authentication)|Developers must bring the Token parameter when logging in to the room to verify the legitimacy of the user.|Basic authentication Token is the basic capability of Token, used for simple permission verification scenarios in business. In most cases, generating this Token is sufficient. |
|[Privilege Authentication Token](/real-time-video-android-java/communication/using-token-authentication)|To further improve security, the room ID and publish stream ID permission bits are opened, which can verify the ID of the logged-in room and the publish stream ID.|General usage scenarios for room ID and publish stream ID permission bits are as follows:<ul><li>Rooms have the distinction between ordinary rooms and member rooms, and it is necessary to control non-member users from logging in to member rooms.</li><li>In voice chat rooms or show live streaming, it is necessary to control the consistency between publishing users and on-mic users to prevent "ghost mic" phenomenon, that is, hearing the voices of non-mic users in the room.</li><li>In speech games such as Werewolf, it is necessary to prevent hackers from cracking the application and using other user IDs to log in to the same room to obtain game progress information for cheating, affecting the game experience of normal users.</li></ul>|


## Prerequisites

<Warning title="Attention">
- Only ZEGO Express SDK version 2.17.0 and above support using Token authentication as described in this document.
- If you have integrated ZEGO Express SDK version before 2.17.0 (using AppSign authentication) and now want to upgrade to version 2.17.0 and use Token authentication, you can learn more about AppSign authentication and Token authentication through the [How to upgrade from AppSign authentication to Token authentication](https://doc-zh.zego.im/faq/token_upgrade) document.
</Warning>


## Process Overview

#### Scenario 1: Login room scenario

When using Token authentication, developers need to generate Token first, and then carry Token to log in to the room. The ZEGO server verifies users carrying Token.

Taking using Token to determine whether users can log in to the room as an example, the usage process is introduced as follows:

<Frame width="512" height="auto" caption=""><img src="https://doc-media.zego.im/sdk-doc/Pics/QuickStart/token_uml.png" /></Frame>

1. The developer's client initiates a request to apply for Token.
2. Generate Token on the developer's server and return it to the developer's client.
3. The developer's client carries the obtained Token and userID, roomID information to log in to the corresponding room.
4. ZEGO SDK will automatically send Token to the ZEGO server for verification.
5. The ZEGO server will return the verification result to the ZEGO SDK.
6. The ZEGO SDK then directly returns the verification result to the developer's client, and clients without permission will fail to log in.

## Token Generation and Usage

This section will introduce in detail how developers generate Token through the server, how to use Token, and how to handle Token expiration.

### 1 Obtain AppID and ServerSecret

Generating Token requires the developer's project's unique identifier AppID and key ServerSecret. Please refer to "Project Information" in [Console - Project Management](/console/project-info) to obtain them.

After developers obtain the AppID and ServerSecret information of the project, they can generate Token on their own server according to actual business needs. The developer's client sends a request to apply for Token to the developer's server, and the developer's server generates Token and returns it to the corresponding client.

<Tab title="Generate basic authentication Token">
When generating a basic authentication Token, pass an empty string in the "payload" field. The specific example code is as follows:

```go
package main
import (
    "fmt"
    "github.com/zegoim/zego_server_assistant/token/go/src/token04"
)
/*
Basic authentication token generation example code
*/
func main() {
    var appId uint32 = 1    // Digital ID issued by Zego, unique identifier for each developer
    userId := "demo"   // User ID
    serverSecret := "fa94dd0f974cf2e293728a526b028271"  // Key for AES encryption when obtaining token
    var effectiveTimeInSeconds int64 = 3600    // Effective duration of token, unit: seconds
    var payload string = ""   // Token business authentication extension, leave empty for basic authentication token
    //Generate token
    token, err := token04.GenerateToken04(appId, userId, serverSecret, effectiveTimeInSeconds, payload)
    if err != nil {
	fmt.Println(err)
	return
    }
    fmt.Println(token)
}
```
</Tab>
<Tab title="Generate privilege authentication Token">
To further improve security, privilege authentication Token opens the room ID and publish stream ID permission bits, which can verify the ID of the logged-in room and the publish stream ID. At this time, the "payload" field needs to be generated according to rules. The permission bit verification rules are as follows:

- Only verify login permission: Verify permissions when logging in, do not verify permissions when publishing stream.
- Only verify publish permission: Verify permissions when publishing stream, do not verify permissions when logging in.
- Verify both login and publish permissions: Verify permissions when logging in and publishing stream.

<Warning title="Attention">
When generating privilege authentication Token, "RoomId" must be passed in.
</Warning>

```go
package main
import (
    "encoding/json"
    "fmt"
    "github.com/zegoim/zego_server_assistant/token/go/src/token04"
)
/*
Privilege authentication token generation example code
*/
//Token business extension: Privilege authentication attributes
type RtcRoomPayLoad struct {
    RoomId       string      `json:"room_id"`        //Room ID (required); used for strong verification of the room ID of the interface
    Privilege    map[int]int `json:"privilege"`      //Permission bit switch list; used for strong verification of operation permissions of the interface
    StreamIdList []string    `json:"stream_id_list"` //Stream list; used for strong verification of the stream ID of the interface; allowed to be empty. If empty, stream ID will not be verified
}
func main() {
    var appId uint32 = 1                               // Digital ID issued by Zego, unique identifier for each developer
    roomId := "demo"                                   // Room ID
    userId := "demo"                                   // User ID
    serverSecret := "fa94dd0f974cf2e293728a526b028271" // Key for AES encryption when obtaining token
    var effectiveTimeInSeconds int64 = 3600            // Effective duration of token, unit: seconds

    //Please refer to github.com/zegoim/zego_server_assistant/token/go/src/token04/token04.go definition
    ////Permission bit definition
    //const (
    //	PrivilegeKeyLogin   = 1 // Login permission bit authentication
    //	PrivilegeKeyPublish = 2 // Publish permission bit authentication
    //)
    ////Permission switch definition
    //const (
    //	PrivilegeEnable     = 1 // Has permission
    //	PrivilegeDisable    = 0 // No permission
    //)
    //Business permission authentication configuration, multiple permission bits can be configured
    privilege := make(map[int]int)
    privilege[token04.PrivilegeKeyLogin] = token04.PrivilegeEnable    // Has room login permission
    privilege[token04.PrivilegeKeyPublish] = token04.PrivilegeDisable // No publish permission
    //Token business extension configuration
    payloadData := &RtcRoomPayLoad{
	RoomId:       roomId,
	Privilege:    privilege,
	StreamIdList: nil,
	}
    payload, err := json.Marshal(payloadData)
    if err != nil {
	fmt.Println(err)
	return
    }
    //Generate token
    token, err := token04.GenerateToken04(appId, userId, serverSecret, effectiveTimeInSeconds, string(payload))
    if err != nil {
	fmt.Println(err)
	return
}
    fmt.Println(token)
}
```

### 3 Use Token

#### Scenario 1: Login room scenario

Users carry the obtained Token and user, roomID information to log in to the corresponding room through the [loginRoom](@loginRoom) interface.

<Warning title="Attention">

The userID used when calling the [loginRoom](@loginRoom) interface to log in to the room must be consistent with the userID used when generating Token in "4.2 Generate Token on server".
</Warning>

```java
String roomID = "xxx" // Room ID to log in to
ZegoUser user = new ZegoUser("xxxx");
ZegoRoomConfig config = new ZegoRoomConfig();
config.token = "xxxxxxxxxx"; // Request from developer's server to obtain
engine.loginRoom(roomID, user, config);
```

If developers need to modify permission bits after logging in to the room, they can also call the [renewToken](@renewToken) interface to update Token. After updating, it will affect the permissions for the next login to certain rooms and publishing certain streams. Previously successful room login and stream publishing will not be affected.


```java
String token = getToken(); // Request from developer's server again to obtain Token;
engine.renewToken(roomID, token);
```

### 4 Token Expiration Handling Mechanism

<Warning title="Attention">

Token expiration may cause issues such as abnormal publish and play streams. Please handle expired Tokens promptly in strict accordance with the following instructions.
</Warning>

#### Scenario 1: Login room scenario

30 seconds before Token expires, the SDK will send a notification through the [onRoomTokenWillExpire](@onRoomTokenWillExpire) callback; when logging in to the room after Token expires, the `1002078` (Token expired) error code will be received through `onDebugError` or `onRoomStateChanged`.

After receiving the Token about to expire callback or receiving the Token expired error code, developers need to obtain a new valid Token from their own server and call the [renewToken](@renewToken) interface provided by the SDK to update the Token.

If you have integrated **ZEGO Express SDK version 2.17.0** or above, when Token expires and the [renewToken](@renewToken) interface is not called to update the Token, when permissions expire:
  - Logged-in users will not be kicked out of the room.
  - Currently successful publish and play streams will not be affected. However, after stopping publishing stream, you cannot publish stream again unless Token is updated.


<Note title="Description">
ZEGO also provides another Token expiration handling method. You can contact ZEGO Technical Support to configure:
  - Logged-in users will be kicked out of the room, and only after updating Token can they log in to the room again.
  - Currently successful publish streams will be stopped.
</Note>


```java
@Override
public void onRoomTokenWillExpire(String roomID, int remainTimeInSecond){
    String token = getToken(); // Request from developer's server again to obtain Token;
    engine.renewToken(roomID, token);
}
```

#### Scenario 2: Range scene module

<Accordion title="Token expiration handling in range scene of 10,000-person audio and video" defaultOpen="false">
30 seconds before Token expires, the SDK will send a notification through the [onSceneTokenWillExpire](@onSceneTokenWillExpire) callback; when logging in to the room after Token expires, the `1002078` (Token expired) error code will be received through `onDebugError` or `onRoomStateChanged`.

After receiving the Token about to expire callback or receiving the Token expired error code, developers need to obtain a new valid Token from their own server and call the [ZegoRangeScene.renewToken](@renewToken-ZegoRangeScene) interface provided by the SDK to update the Token. If not handled, the Token expiration handling mechanism is as follows:

- Logged-in users will not be kicked out of the scene.
- Currently successful publish and play streams will not be affected, but will affect the user's next publish and play stream operations.



```java
public void onSceneTokenWillExpire(ZegoRangeScene rangeScene, int remainTimeInSecond) {
    super.onSceneTokenWillExpire(rangeScene, remainTimeInSecond);
    String token = getToken(); // Request from developer's server again to obtain Token;
    // Please switch threads to call renewToken
    rangeScene.renewToken(token);
}
```
</Accordion>
