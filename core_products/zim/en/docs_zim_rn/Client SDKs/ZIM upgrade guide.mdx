# ZIM Upgrade Guide

---

This document provides notes and important information for upgrading the ZIM React Native SDK.

## Upgrade Guide for 2.19.0

<Warning title="Warning">

Starting from version 2.19.0, several APIs have significant changes. Please read the following guide carefully if you are upgrading from an earlier version to 2.19.0.
</Warning>

### downloadMediaFile and Related Callbacks

The old `downloadMediaFile` API is deprecated. Please use the new [downloadMediaFile](@downloadMediaFile) API with the same name instead. The new version of [downloadMediaFile](@downloadMediaFile) adds a `config` parameter, which can be used to specify downloading a specific media item in a composite message.

In [ZIMMediaDownloadingProgress](@ZIMMediaDownloadingProgress) and [ZIMMediaDownloadedResult](@-ZIMMediaDownloadedResult), the `message` parameter's type has changed from `ZIMMediaMessage` to `ZIMMessage` to support composite messages. TypeScript developers should adjust their code according to IDE compilation error hints.

<CodeGroup>
```typescript Usage in 2.19.0 (downloading a media item in a multi-message) {14-17,19,29}
// Suppose multipleMessage.messageInfoList[0] is a text message, multipleMessage.messageInfoList[1] is an image message
const multipleMessage: ZIMMessage = {
    type: 10,
    messageInfoList: [
        { type: 1, message: "Hello, World!" },
        { type: 11, fileLocalPath: '' }
    ]
}

const config: ZIMMediaDownloadConfig = {
    // Download the image message
    messageInfoIndex: 1
}

zim.downloadMediaFile(multipleMessage, 1, config, (message: ZIMMessage, currentFileSize: number, totalFileSize: number) => {
    // Download progress
    // Developers need to check message type and convert to the corresponding message type
    if (message.type === 10) {
        const multipleMessage: ZIMMultipleMessage = message as ZIMMultipleMessage
        // Handle composite message
    }
    // Handle other message types
    ......

}).then((res: ZIMMediaDownloadedResult) => {
    // Download completed
    // Developers need to check message type and convert to the corresponding message type
    if (res.message.type === 10) {
        const multipleMessage = res.message as ZIMMultipleMessage
        // Handle composite message
    }
    // Handle other message types
    ......
}).catch((errorInfo) => {
    // Download failed
})
```

```typescript Usage in 2.19.0 (downloading a media message) {6,15}
const imageMessage: ZIMMessage = {
    type: 11,
    fileLocalPath: ''
}

zim.downloadMediaFile(imageMessage, 1, (message: ZIMMessage, currentFileSize: number, totalFileSize: number) => {
    // Download progress
    // Developers need to check message type and convert to the corresponding message type
    if (message.type === 11) {
        const imageMessage: ZIMImageMessage = message as ZIMImageMessage
        // Handle image message
    }
    // Handle other message types
    ......
}).then((res: ZIMMediaDownloadedResult) => {
    // Download completed
    // Developers need to check message type and convert to the corresponding message type
    if (res.message.type === 11) {
        const imageMessage = res.message as ZIMImageMessage
        // Handle image message
    }
    // Handle other message types
    ......
}).catch((errorInfo) => {
    // Download failed
})
```

```typescript Usage in previous versions {6,8}
const imageMessage: ZIMMessage = {
    type: 11,
    fileLocalPath: ''
}

zim.downloadMediaFile(imageMessage, 1, (message: ZIMMediaMessage, currentFileSize: number, totalFileSize: number) => {
    // Download progress
}).then((res: ZIMMediaDownloadedResult) => {
    // Download completed
}).catch((errorInfo) => {
    // Download failed
})
```
</CodeGroup>

### sendMessage and Related Callbacks

Since version 2.19.0, sending media messages should use the [sendMessage](@sendMessage) API. The `sendMediaMessage` API is deprecated to unify message sending and facilitate future extensibility.

In [ZIMMessageSendNotification](@-ZIMMessageSendNotification), the [onMediaUploadingProgress](@onMediaUploadingProgress-ZIMMessageSendNotification) callback’s `message` parameter type has changed from `ZIMMessage` to `ZIMMediaMessage`. This ensures only media messages trigger the progress callback. TypeScript developers should adjust their code as per IDE compilation error prompts. (Currently, only developers using TypeScript and the [replyMessage](@replyMessage) API will encounter related compile errors.)

<CodeGroup>
```typescript Usage in 2.19.0 {10,14,19}
const imageMessage: ZIMMessage = {
    type: 11,
    fileLocalPath: ''
}

const config: ZIMMessageSendConfig = {
    priority: 1
}

const notification: ZIMMessageSendNotification = {
    onMessageAttached: (message: ZIMMessage) => {
        // Callback where developers can perform business logic before the message is sent
    },
    onMessageUploadingProgress: (message: ZIMMediaMessage, currentFileSize: number, totalFileSize: number) => {
        // Media upload progress
    }
}

zim.sendMessage(imageMessage, "TO_CONVERSATION_ID", 0, config, notification)
    .then((res: ZIMMessageSentResult) => {
        // Message sent result
    }).catch((errorInfo) => {
        // Message sending failed
    })
```

```typescript Usage in previous versions {10,14,19}
const imageMessage: ZIMMessage = {
    type: 11,
    fileLocalPath: ''
}

const config: ZIMMessageSendConfig = {
    priority: 1
}

const notification: ZIMMediaMessageSendNotification = {
    onMessageAttached: (message: ZIMMediaMessage) => {
        // Callback where developers can perform business logic before the message is sent
    },
    onMessageUploadingProgress: (message: ZIMMediaMessage, currentFileSize: number, totalFileSize: number) => {
        // Media upload progress
    }
}

zim.sendMediaMessage(imageMessage, "TO_CONVERSATION_ID", 0, config, notification)
    .then((res: ZIMMessageSentResult) => {
        // Message sent result
    }).catch((errorInfo) => {
        // Message sending failed
    })
```
</CodeGroup>


## Upgrade Guide for 2.18.0

<Warning title="Warning">

Starting from version 2.18.0, several APIs have significant changes. Please read the following guide carefully if you are upgrading from an earlier version to 2.18.0.
</Warning>

### Peer Message Receive Callback

The original single chat message receive callback `receivePeerMessage` is deprecated. Please use [peerMessageReceived](@peerMessageReceived) instead.

The new callback supports:
- Receiving online peer-to-peer messages while the user is online.
- After the user re-logs in to ZIM SDK, receiving all offline peer messages sent within the last 7 days.

```typescript
// New API
peerMessageReceived: (zim: ZIM, data: ZIMEventOfConversationMessageReceivedResult) => void;

// Old API
receivePeerMessage: (zim: ZIM, data: ZIMEventOfReceiveConversationMessageResult) => void;
```

### Room Message Receive Callback

The original room message receive callback `receiveRoomMessage` is deprecated. Please use [roomMessageReceived](@roomMessageReceived) instead.

The new callback supports:
- Receiving online room messages while the user is online.
- Once back online, if the user is still in the room, receiving all offline room messages sent while offline.

```typescript
// New API
roomMessageReceived: (zim: ZIM, data: ZIMEventOfConversationMessageReceivedResult) => void;

// Old API
receiveRoomMessage: (zim: ZIM, data: ZIMEventOfReceiveConversationMessageResult) => void;
```

### Group Message Receive Callback

The original group message receive callback `receiveGroupMessage` is deprecated. Please use [groupMessageReceived](@groupMessageReceived) instead.
 
The new callback supports:
- Receiving online group messages while the user is online.
- After the user re-logs in to ZIM SDK, receiving all offline group messages sent within the past 7 days.

```typescript
// New API
groupMessageReceived: (zim: ZIM, data: ZIMEventOfConversationMessageReceivedResult) => void;

    
// Old API
receiveGroupMessage: (zim: ZIM, data: ZIMEventOfReceiveConversationMessageResult) => void;
```

## Upgrade Guide for 2.16.0

<Warning title="Warning">

Starting from version 2.16.0, several APIs have significant changes. Please read the following guide carefully if you are upgrading from an earlier version to 2.16.0.
</Warning>

#### callCancel

<Note title="Note">

The following changes only apply to **Advanced Mode** call invitations.
</Note>

In the new [callCancel](@callCancel) API, if the userIDs parameter contains a userID, the API will cancel the invitation for that specific callee only. If the userIDs parameter is empty, the API will cancel invitations for all callees.

In contrast, the old version of `callCancel`, regardless of whether userIDs is empty or not, would cancel invitations for all callees.

Since the old ZIM SDK does not support cancellation for individual callees, if you need to keep the old cancellation logic and also use the new single-cancel feature, please isolate call logic between old and new ZIM versions.

<CodeGroup>
```typescript Usage in 2.16.0
// Cancel invitations for userIdA and userIdB only
const callID = 'xxxx';
const invitees = ['userIdA', 'userIdB'];  // List of invitee userIDs
const config: ZIMCallCancelConfig  = { extendedData: 'xxxx' }; 
zim.callCancel(invitees, callID, config)
    .then((res: ZIMCallCancelSentResult) => {
        // Success
    })
    .catch((err: ZIMError) => {
        // Failure
    })

// Cancel the entire call invitation; works only if none of the callees have accepted
const callID = 'xxxx';
const invitees = [];  // List of invitee userIDs
const config: ZIMCallCancelConfig = { extendedData: 'xxxx' }; 
zim.callCancel(invitees, callID, config)
    .then((res: ZIMCallCancelSentResult) => {
        // Success
    })
    .catch((err: ZIMError) => {
        // Failure
    })
```
```typescript Usage in previous versions
// Cancels the entire call invitation, regardless of whether userIDs is empty or contains userIDs. 
// Works only if none of the callees have accepted.
const callID = 'xxxx';
const invitees = ['userIdA', 'userIdB'];  // List of invitee userIDs
const config: ZIMCallCancelConfig = { extendedData: 'xxxx' }; 
zim.callCancel(invitees, callID, config)
    .then((res: ZIMCallCancelSentResult) => {
        // Success
    })
    .catch((err: ZIMError) => {
        // Failure
    })

// Cancel the entire call invitation; works only if none of the callees have accepted
const callID = 'xxxx';
const invitees = [];  // List of invitee userIDs
const config: ZIMCallCancelConfig = { extendedData: 'xxxx' }; 
zim.callCancel(invitees, callID, config)
    .then((res: ZIMCallCancelSentResult) => {
        // Success
    })
    .catch((err: ZIMError) => {
        // Failure
    })
```
</CodeGroup>

## Upgrade Guide for 3.0.0

<Warning title="Warning">

This version (3.0.0) removes APIs that have been deprecated for over 1 year, along with related enum values and fields, and adds deprecation markers to some APIs. The React Native SDK involves changes at both the TypeScript interface layer and the native layer (iOS/Android). Please refer to the following instructions to complete the migration.
</Warning>

### Removed deprecated APIs

#### ZIM initialization API (create)

The deprecated `create(appID: number)` API has been removed. Please use the [create](@create) API that accepts [ZIMAppConfig](@ZIMAppConfig) instead.

<CodeGroup>
```typescript New version usage
const appConfig: ZIMAppConfig = {
    appID: 12345678,
    appSign: 'appSign',
};
ZIM.create(appConfig);
```

```typescript Old version usage
// Old API has been removed
ZIM.create(12345678);
```
</CodeGroup>

#### ZIM login API (login)

The deprecated `login(userInfo: ZIMUserInfo, token: string)` API has been removed. Please use the [login](@login) API that accepts `userID` and [ZIMLoginConfig](@ZIMLoginConfig) instead.

<CodeGroup>
```typescript New version usage
const loginConfig: ZIMLoginConfig = {
    userName: 'userName',
    token: '', // Fill in the token if using token authentication
    isOfflineLogin: false,
};
await zim.login('userID', loginConfig);
```

```typescript Old version usage
// Old API has been removed
const userInfo: ZIMUserInfo = { userID: 'userID', userName: 'userName' };
await zim.login(userInfo, '');
```
</CodeGroup>

#### ZIM message sending API (sendMessage)

The deprecated `sendPeerMessage`, `sendRoomMessage`, `sendGroupMessage`, and `sendMediaMessage` APIs have been removed. Please use the [sendMessage](@sendMessage) API with the `notification` parameter instead.

<CodeGroup>
```typescript New version usage
const textMessage: ZIMTextMessage = {
    type: 1,
    message: 'Hello',
};
const config: ZIMMessageSendConfig = {
    priority: 1,
};
const notification: ZIMMessageSendNotification = {
    onMessageAttached: (message: ZIMMessage) => {
        // Business logic before message is sent
    },
};
zim.sendMessage(textMessage, 'toConversationID', 0, config, notification)
    .then((res: ZIMMessageSentResult) => {
        // Message sent result
    })
    .catch((errorInfo) => {
        // Message sent failed
    });
```

```typescript Old version usage
// The following APIs have been removed, please migrate to sendMessage
zim.sendPeerMessage(textMessage, 'toUserID', config);
zim.sendGroupMessage(textMessage, 'toGroupID', config);
zim.sendMediaMessage(imageMessage, 'toConversationID', 0, config, notification);
```
</CodeGroup>

#### ZIM media file download API (downloadMediaFile)

The deprecated `downloadMediaFile` API (without `config` parameter) has been removed. Please use the new [downloadMediaFile](@downloadMediaFile) API with the [ZIMMediaDownloadConfig](@ZIMMediaDownloadConfig) parameter instead.

<CodeGroup>
```typescript New version usage
const config: ZIMMediaDownloadConfig = {};
zim.downloadMediaFile(imageMessage, 1, config, (message: ZIMMessage, currentFileSize: number, totalFileSize: number) => {
    // Download progress
}).then((res: ZIMMediaDownloadedResult) => {
    // Download complete
}).catch((errorInfo) => {
    // Download failed
});
```

```typescript Old version usage
// Old API has been removed, please migrate to the new version with config parameter
zim.downloadMediaFile(imageMessage, 1, (message: ZIMMediaMessage, currentFileSize: number, totalFileSize: number) => {
    // Download progress
}).then((res: ZIMMediaDownloadedResult) => {
    // Download complete
});
```
</CodeGroup>

#### ZIM message receive callback API

The `receivePeerMessage`, `receiveRoomMessage`, and `receiveGroupMessage` events deprecated in 2.18.0 have been officially removed in this version. It is recommended to migrate to [messageReceived](@messageReceived).

<CodeGroup>
```typescript New version usage
zim.on('messageReceived', (zim: ZIM, result: ZIMMessageReceivedEventResult) => {
    const { messageList, info, conversationID, conversationType } = result;
    // Handle messages for different conversation types based on conversationType
});
```

```typescript Old version usage (removed)
zim.on('receivePeerMessage', (zim: ZIM, data: ZIMEventOfReceiveConversationMessageResult) => {});
zim.on('receiveRoomMessage', (zim: ZIM, data: ZIMEventOfReceiveConversationMessageResult) => {});
zim.on('receiveGroupMessage', (zim: ZIM, data: ZIMEventOfReceiveConversationMessageResult) => {});
```
</CodeGroup>

#### ZIM call invitation callback APIs

The old `callInvitationRejected`, `callInvitationAccepted`, and `callInviteesAnsweredTimeout` events have been removed. Please use the [callUserStateChanged](@callUserStateChanged) event instead. The callback parameters of the `callInvitationTimeout` event have also been updated, adding a `ZIMCallInvitationTimeoutInfo` parameter.

<CodeGroup>
```typescript New version usage
zim.on('callUserStateChanged', (zim: ZIM, result: ZIMCallUserStateChangedEventResult) => {
    const { callUserList, callID } = result;
    // callUserList contains the list of users whose status changed
    // You can uniformly handle accept, reject, timeout, and other status changes
});
```

```typescript Old version usage
// Old callbacks have been removed, please migrate to callUserStateChanged
zim.on('callInvitationRejected', (zim: ZIM, data: ZIMEventOfCallInvitationRejectedResult) => {});
zim.on('callInvitationAccepted', (zim: ZIM, data: ZIMEventOfCallInvitationAcceptedResult) => {});
zim.on('callInviteesAnsweredTimeout', (zim: ZIM, data: ZIMEventOfCallInviteesAnsweredTimeoutResult) => {});
```
</CodeGroup>

#### importLocalMessages / exportLocalMessages APIs removed

The `importLocalMessages` and `exportLocalMessages` APIs have been temporarily removed in version 3.0.0, and there is no replacement API at this time. This functionality will be re-enabled in future versions. Please follow the SDK release notes for updates.

### Newly deprecated APIs (recommended to migrate)

#### ZIM message receive callback APIs

The `peerMessageReceived`, `roomMessageReceived`, and `groupMessageReceived` events have been marked as deprecated in this version. They can still be used normally at present, but it is recommended to migrate to [messageReceived](@messageReceived) as soon as possible.

<CodeGroup>
```typescript Recommended (new event)
zim.on('messageReceived', (zim: ZIM, result: ZIMMessageReceivedEventResult) => {
    const { messageList, info, conversationID, conversationType } = result;
    // Handle messages for different conversation types based on conversationType
});
```

```typescript Deprecated event (still usable, recommended to migrate)
zim.on('peerMessageReceived', (zim: ZIM, data: ZIMConversationMessageReceivedEventResult) => {});
zim.on('roomMessageReceived', (zim: ZIM, data: ZIMConversationMessageReceivedEventResult) => {});
zim.on('groupMessageReceived', (zim: ZIM, data: ZIMConversationMessageReceivedEventResult) => {});
```
</CodeGroup>

#### ZIMGroupConversation deprecated

`ZIMGroupConversation` has been deprecated in 3.0.0. The `isDisabled` and `mutedExpiredTime` fields should be replaced with the corresponding properties of the base class `ZIMConversation`:

| ZIMGroupConversation (deprecated) | ZIMConversation (replacement property) |
|---|---|
| `isDisabled` | `isConversationDisabled` |
| `mutedExpiredTime` | `selfMutedExpiredTime` |

<CodeGroup>
```typescript New version usage
const isDisabled: boolean = conversation.isConversationDisabled;
const mutedExpiredTime: number = conversation.selfMutedExpiredTime;
```

```typescript Old version usage (deprecated)
const groupConversation = conversation as ZIMGroupConversation;
const isDisabled: boolean = groupConversation.isDisabled;
const mutedExpiredTime: number = groupConversation.mutedExpiredTime;
```
</CodeGroup>

### Type interface name changes

All TypeScript interfaces prefixed with `ZIMEventOf` in `ZIMEventHandler.ts` have been renamed following a unified convention:

**Naming convention**: `ZIMEventOf<Name>Result` → `ZIM<Name>EventResult`

**Typical examples**:

| Old type name | New type name |
|---|---|
| `ZIMEventOfConversationMessageReceivedResult` | `ZIMConversationMessageReceivedEventResult` |
| `ZIMEventOfCallInvitationRejectedResult` | `ZIMCallInvitationRejectedEventResult` |
| `ZIMEventOfCallInvitationAcceptedResult` | `ZIMCallInvitationAcceptedEventResult` |
| `ZIMEventOfCallInviteesAnsweredTimeoutResult` | `ZIMCallInviteesAnsweredTimeoutEventResult` |

<Note title="Scope of impact">

This change **is transparent to most developers and requires no code changes**. The reason: the callback parameter types of `zim.on(eventName, callback)` are automatically inferred by TypeScript based on the event name, without the need to explicitly reference these interface names.

Only when you have **explicitly annotated** `ZIMEventOf*` type names in your code will TypeScript compilation errors occur. Simply complete the type name replacement according to the table above.
</Note>

### setRoomMembersAttributes behavior change

Starting from version 3.0.0, when setting room member attributes via [setRoomMembersAttributes](@setRoomMembersAttributes), if `isDeleteAfterOwnerLeft` is `false`, the room member attributes will not be deleted when the user leaves the room. In versions 2.x.x, attributes were first deleted when the user left the room and then restored when the user returned.

- - -
