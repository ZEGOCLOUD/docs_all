# 智能体自定义控制

智能体自定义控制允许客户端通过 **ZEGO Express SDK** 房间信令直接发起智能体实例控制请求，无需依赖服务端 API 调用。

通过该功能，开发者可以在客户端侧实现以下控制能力：

- **主动调用 TTS**：让智能体用语音朗读一段文本，适合播报固定话术（如欢迎语、提醒等）。
- **主动调用 LLM**：给智能体一段文本，让其基于 LLM 推理并回复，适合需要结合上下文的主动发言场景。
- **打断智能体**：立即打断智能体当前正在进行的 TTS/LLM 流程。
- **控制聆听**：在对讲机模式下，控制智能体开始或停止聆听指定用户。

典型使用场景包括：
- 播报欢迎语、超时提醒等固定话术。
- 冷场时主动发起话题，提升交互体验。
- 触发关键节点时打断智能体并立刻播报重要内容。

适用版本：v2.13.0 及以上。

## 前提条件

已参考[快速开始](./../quick-start.mdx)完成基本流程。

## 使用方式

### 方式一：通过开源组件实现（推荐）

使用 [ZEGO AI Agent Action 客户端套件](https://github.com/ZEGOCLOUD/ai_agent_quick_start/tree/master/agentaction)，可快速集成智能体自定义控制能力。

该套件支持 Web、Android、iOS Swift、iOS Objective-C、Flutter 五大平台，对外统一暴露以下方法：

- `sendAgentInstanceTTS`
- `sendAgentInstanceLLM`
- `interruptAgentInstance`
- `startListening`
- `stopListening`

#### 集成步骤

1. 将目标平台下的 `agentaction/` 目录拷贝到自己的项目中。
2. 创建 Client 实例。
3. 在 Sender 中调用 Express `callExperimentalAPI`，将套件生成的 `formatedJson` 透传入。
4. 在 `onRecvExperimentalAPI` 回调中，将响应内容转交给套件的 `handleRoomChannelMessage` 方法。

|参数名-150px | 描述 | 示例值|
| --- | --- | --- |
| roomId | 业务房间 ID，与 ZEGO 音视频房间 ID 一致。 | `room_001` |

| timeoutMs | 单次请求默认超时（ms），默认 5000。 | `5000` |
| sender | 业务实现的发送回调，把套件组装好的 `formatedJson` 透传给 Express `callExperimentalAPI`。 | 无固定取值，见上方各端 sender 实现 |
| onResponse | 全局响应回调，每次收到 PaaS 响应时触发（可选）。 | 无固定取值，见上方各端 onResponse 实现 |
| onError | 全局错误回调，发送失败 / 超时 / 主动取消时触发（可选）。 | 无固定取值，见上方各端 onError 实现 |

### 方式二：自定义实现

如果不想使用开源套件，也可以自行实现智能体控制。其原理是：客户端通过 ZEGO Express SDK 的 `callExperimentalAPI` 发送房间通道消息（msg_type=20）到服务端，服务端处理后通过房间通道消息（msg_type=22）返回响应，客户端通过 `onRecvExperimentalAPI` 接收响应。

#### 调用 callExperimentalAPI 接口发送指令

以下展示各平台使用 ZEGO Express SDK 发送控制指令。完整可运行的示例代码请参考 [ai_agent_quick_start (agentaction)](https://github.com/ZEGOCLOUD/ai_agent_quick_start/tree/master/agentaction)。

使用 Express SDK 的 `callExperimentalAPI` 发送房间通道消息（msg_type=20），各平台调用方式如下：

<Tabs>
<Tab title="Android / iOS / Flutter">

Android、iOS 和 Flutter 端的 `callExperimentalAPI` 调用方式一致，`method` 参数使用 `liveroom.room.send_room_channel_message`。传入 JSON 字符串格式的请求体：

```json
{
    "method": "liveroom.room.send_room_channel_message",
    "params": {
        "room_id": "room_001",
        "msg_type": 20,
        "msg_content": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"Text\":\"你好\"}}",
        "user_list": ["<AgentUserId>"],
        "seq": 1
    }
}
```
</Tab>
<Tab title="Web">

Web 端 `method` 参数使用 `sendRoomChannelMessage` ，字段命名采用驼峰格式。传入 JSON 字符串格式的请求体：

<Warning title="注意"> Web 端发送没有回调，发送成功与否通过 `callExperimentalAPI` 是否抛错来判断，业务结果需等待智能体回包或超时。</Warning>

```json
{
    "method": "sendRoomChannelMessage",
    "params": {
        "roomID": "room_001",
        "msgType": 20,
        "msgContent": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"Text\":\"你好\"}}",
        "toUserIDList": ["<agent_user_id>"],
        "seq": 1
    }
}
```
</Tab>
</Tabs>

##### params.user_list 参数

Android、iOS和 Flutter 端的 `user_list`，Web 端的 `toUserIDList`，均为目标智能体 userID（只传一个）。

<Warning title="注意">
数字人通话场景下，传递给 user_list（toUserIDList） 的目标智能体 userID 与普通 1V1 智能体场景不同，客户端必须按以下规则传递：

| 场景 | 目标智能体 userID 取值 |
| --- | --- |
| 普通智能体（通过[创建智能体实例](/aiagent-server/api-reference/agent-instance-management/create-agent-instance)创建） | 与接口请求参数的 `RTC.AgentUserId` 字段的值相同。比如 `agent_123456` |
| 数字人智能体（通过[创建数字人 Agent 实例](/aiagent-server/api-reference/agent-instance-management/create-digital-human-agent-instance)创建） | 按照`ai_agent_<AgentInstanceId>`模板生成。AgentInstanceId就是响应参数中的 Data.AgentInstanceId。比如 `ai_agent_1912122918452641792` |


</Warning>

##### params.msg_content 参数

其中 `msg_content` 为 JSON 字符串，包含以下字段：

|字段-100px | 类型 | 描述|
| --- | --- | --- |
| Action | String | 操作类型，如 `SendAgentInstanceTTS`、`SendAgentInstanceLLM`、`InterruptAgentInstance`、`StartListening`、`StopListening`。请参考 [Action](#action) |
| Seq | String | 业务链路追踪标识，由客户端生成，需保证同房间多用户/多端场景下唯一。推荐格式 `user_id:device_id:local_seq`。在 onRecvExperimentalAPI 将原样返回。|
| Params | Object | 各 Action 对应的参数。请参考 [Params](#params) |


###### Action

callExperimentalAPI 接口的 msg_content 参数中，JSON 字符串的 Params 字段如下。客户端通过房间信令支持以下 5 类控制能力：

| Action | 能力描述 | 对应服务端接口 |
| --- | --- | --- |
| `SendAgentInstanceTTS` | 主动调用 TTS，让智能体朗读一段文本。 | [自定义调用 TTS](/aiagent-server/api-reference/agent-instance-control/send-agent-instance-tts) |
| `SendAgentInstanceLLM` | 主动调用 LLM，让智能体基于文本推理并回复。 | [自定义调用 LLM](/aiagent-server/api-reference/agent-instance-control/send-agent-instance-llm) |
| `InterruptAgentInstance` | 打断智能体，停止当前 TTS/LLM 流程。 | [打断智能体实例](/aiagent-server/api-reference/agent-instance-control/interrupt-agent-instance) |
| `StartListening` | 智能体开始聆听指定用户。 | [智能体开始聆听](/aiagent-server/api-reference/agent-instance-control/start-listening) |
| `StopListening` | 智能体结束聆听指定用户。 | [智能体结束聆听](/aiagent-server/api-reference/agent-instance-control/stop-listening) |

###### Params

callExperimentalAPI 接口的 msg_content 参数中，JSON 字符串的 Params 字段每个 Action 都会对应不同的选项。以下每个 Tab 对应不同的 Action 的 Params 说明：

<Tabs titleSize="p">
<Tab title="SendAgentInstanceTTS">

主动调用 TTS，让智能体用语音朗读一段文本。

**参数**

| 参数 | 类型 | 是否必选 | 描述 |
| --- | --- | --- | --- |
| Text | String | 是 | 用于 TTS 的文本内容，最大不超过 300 个字符。 |
| AddHistory | Boolean | 否 | 是否将文本消息记录在会话消息历史之中，作为上下文输入给 LLM。默认值为 `true`。 |
| Priority | String | 否 | 任务优先级，默认为 `Medium`。可选值：`Low`、`Medium`、`High`。 |
| SamePriorityOption | String | 否 | 相同优先级时的打断策略，默认为 `ClearAndInterrupt`。可选值：`ClearAndInterrupt`、`Enqueue`。 |
| InterruptMode | Number | 否 | 打断模式。 |
| EnqueueUserSpeech | Boolean | 否 | 是否排队用户语音。 |

**请求示例**

```json
{
    "Action": "SendAgentInstanceTTS",
    "Seq": "client_A:web_01:1",
    "Params": {
        "Text": "你好",
        "AddHistory": true,
        "Priority": "Medium",
        "SamePriorityOption": "ClearAndInterrupt"
    }
}
```

</Tab>
<Tab title="SendAgentInstanceLLM">

主动调用 LLM，让智能体基于文本推理并回复。

**参数**

| 参数 | 类型 | 是否必选 | 描述 |
| --- | --- | --- | --- |
| Text | String | 是 | 发送给 LLM 服务的文本内容。 |

#### 监听 onRecvExperimentalAPI 接收响应

服务端处理完成后，通过房间通道消息（msg_type=22）下发响应，客户端在 ZEGO Express SDK 的 `onRecvExperimentalAPI` 回调中接收。回调内容为 JSON 字符串（原生端）或对象（Web 端），其中 `msg_content`/`msgContent` 本身又是一层 JSON 字符串，需**二次解析**才能得到响应体，最后通过响应体中的 `Seq` 关联回原始请求。

<Tabs>
<Tab title="Android / iOS / Flutter">

原生端在创建 `ZegoExpressEngine` 时实现 `onRecvExperimentalAPI(String content)` 回调，`content` 为 JSON 字符串（字段采用 snake_case），结构与发送时一致：

```json
{
    "method": "liveroom.room.on_recive_room_channel_message",
    "params": {
        "room_id": "room_001",
        "msg_type": 22,
        "msg_content": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Code\":0,\"Message\":\"success\",\"RequestId\":\"8916619579948490296\",\"Data\":{\"Round\":1481651956}}"
    }
}
```

以 Android（Java）为例，从 `params.msg_content` 二次解析出响应体：

```java
ZegoExpressEngine.createEngine(profile, appContext, new IZegoEventHandler() {
    @Override
    public void onRecvExperimentalAPI(String content) {
        try {
            JSONObject data = new JSONObject(content);
            // 仅处理业务响应回调（msg_type=22）
            if (!"liveroom.room.on_recive_room_channel_message".equals(data.optString("method"))) {
                return;
            }
            JSONObject params = data.optJSONObject("params");
            if (params == null || params.optInt("msg_type", -1) != 22) {
                return;
            }
            // msg_content 为嵌套 JSON 字符串，需二次解析得到响应体
            JSONObject resp = new JSONObject(params.optString("msg_content"));
            String seq = resp.optString("Seq");
            int code = resp.optInt("Code");
            // 按 seq 关联回原始请求；code == 0 表示成功，其余为业务错误码
        } catch (JSONException e) {
            e.printStackTrace();
        }
    }
});
```

iOS（Objective-C / Swift）与 Flutter 的解析逻辑一致：先判断 `method` 为 `liveroom.room.on_recive_room_channel_message`，再从 `params.msg_content` 二次解析出响应体，按 `Seq` 关联请求。

</Tab>
<Tab title="Web">

Web 端通过 `zg.on('recvExperimentalAPI', payload => {...})` 事件接收，`payload` 为对象（字段采用 camelCase）。`payload.msgContent` 同样是嵌套 JSON 字符串，需二次解析。部分 Web Express 版本需先显式开启房间通道消息接收：

```json
{
    "method": "onRecvRoomChannelMessage",
    "content": {
        "roomID": "room_001",
        "msgType": 22,
        "msgContent": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Code\":0,\"Message\":\"success\",\"RequestId\":\"8916619579948490296\",\"Data\":{\"Round\":1481651956}}"
    }
}
```

```javascript
// 部分 Web Express 版本需显式开启房间通道消息接收
zg.callExperimentalAPI({ method: 'onRecvRoomChannelMessage', params: {} });

// 监听实验性 API 回调，解析响应
zg.on('recvExperimentalAPI', (payload) => {
    if (payload.method !== 'onRecvRoomChannelMessage') {
        return;
    }
    const body = payload.content || payload.params;
    if (body.msgType !== 22) {
        return;
    }
    // msgContent 为嵌套 JSON 字符串，需二次解析得到响应体
    const resp = JSON.parse(body.msgContent);
    const { Seq, Code, Message } = resp;
    // 按 Seq 关联回原始请求；Code === 0 表示成功，其余为业务错误码
});
```

### 注意事项

#### Web 端与原生端协议差异

| 差异点 | Android / iOS 原生端 | Web 端 |
| --- | --- | --- |
| 发送方法名 | `liveroom.room.send_room_channel_message` | `sendRoomChannelMessage` |
| 接收回调方法名 | `liveroom.room.on_recive_room_channel_message` | `onRecvRoomChannelMessage` |
| 房间 ID 字段 | `room_id` | `roomID` |
| 接收方列表字段 | `user_list` | `toUserIDList` |

#### 链路追踪与重发去重

- `Seq` 由客户端按 `user_id:device_id:local_seq` 格式生成。
- 客户端按 `Seq` 维护请求表，对同一 `Seq` 的多个响应只处理第一个有效响应。
- 业务幂等由服务端处理，客户端无需关心。
