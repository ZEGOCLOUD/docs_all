# 智能体自定义控制

智能体自定义控制允许客户端通过 **ZEGO Express SDK** 房间信令直接发起智能体实例控制请求，无需依赖服务端 API 调用。

通过该功能，开发者可以在客户端侧实现以下控制能力：

- **主动调用 TTS**：让智能体用语音朗读一段文本，适合播报固定话术（如欢迎语、提醒等）。
- **主动调用 LLM**：给智能体一段文本，让其基于 LLM 推理并回复，适合需要结合上下文的主动发言场景。
- **打断智能体**：立即打断智能体当前正在进行的 TTS/LLM 流程。
- **控制聆听**：在对讲机模式下，控制智能体开始或停止聆听指定用户。

典型使用场景包括：
- 播报欢迎语、超时提醒等固定话术。
- 冷场时主动发起话题，提升交互体验。
- 触发关键节点时打断智能体并立刻播报重要内容。

适用版本：v2.13.0 及以上。

## 前提条件

已参考[快速开始](./../quick-start.mdx)完成基本流程。

## 使用方式

### 方式一：通过开源组件实现（推荐）

使用 [ZEGO AI Agent Action 客户端套件](https://github.com/ZEGOCLOUD/ai_agent_quick_start/tree/master/agentaction)，可快速集成智能体自定义控制能力。

该套件支持 Web、Android、iOS Swift、iOS Objective-C、Flutter 五大平台，对外统一暴露以下方法：

- `sendAgentInstanceTTS`
- `sendAgentInstanceLLM`
- `interruptAgentInstance`
- `startListening`
- `stopListening`

#### 集成步骤

1. 将目标平台下的 `agentaction/` 目录拷贝到自己的项目中。
2. 创建 Client 实例。
3. 在 Sender 中调用 Express `callExperimentalAPI`，将套件生成的 `formatedJson` 透传入。
4. 在 `onRecvExperimentalAPI` 回调中，将响应内容转交给套件的 `handleRoomChannelMessage` 方法。

### 方式二：自定义实现

如果不想使用开源套件，也可以自行实现智能体控制。其原理是：客户端通过 ZEGO Express SDK 的 `callExperimentalAPI` 发送房间通道消息（msg_type=20）到服务端，服务端处理后通过房间通道消息（msg_type=22）返回响应。

#### 消息协议

客户端通过房间信令支持以下 5 类控制能力：

| Action | 能力描述 | 对应服务端接口 |
| --- | --- | --- |
| `SendAgentInstanceTTS` | 主动调用 TTS，让智能体朗读一段文本。 | [自定义调用 TTS](/aiagent-server/api-reference/agent-instance-control/send-agent-instance-tts) |
| `SendAgentInstanceLLM` | 主动调用 LLM，让智能体基于文本推理并回复。 | [自定义调用 LLM](/aiagent-server/api-reference/agent-instance-control/send-agent-instance-llm) |
| `InterruptAgentInstance` | 打断智能体，停止当前 TTS/LLM 流程。 | [打断智能体实例](/aiagent-server/api-reference/agent-instance-control/interrupt-agent-instance) |
| `StartListening` | 智能体开始聆听指定用户。 | [智能体开始聆听](/aiagent-server/api-reference/agent-instance-control/start-listening) |
| `StopListening` | 智能体结束聆听指定用户。 | [智能体结束聆听](/aiagent-server/api-reference/agent-instance-control/stop-listening) |

**消息类型：**

| 消息类型 | 取值 | 方向 | 说明 |
| --- | --- | --- | --- |
| 请求 | `20` | 客户端 → 服务端 | 客户端发起控制请求。 |
| 响应 | `22` | 服务端 → 客户端 | 服务端返回处理响应。 |

**消息体结构（msg_content）：**

```json
{
    "Action": "SendAgentInstanceTTS",
    "Seq": "client_A:web_01:1",
    "Params": {
        "Key": "Value"
    }
}
```

| 字段 | 类型 | 是否必选 | 描述 |
| --- | --- | --- | --- |
| Action | String | 是 | 控制动作名，对应上表中的 5 类 Action。 |
| Seq | String | 是 | 业务链路追踪标识，由客户端生成，需保证同房间多用户/多端场景下唯一。推荐格式 `user_id:device_id:local_seq`。 |
| Params | Object | 是 | 不同 Action 的具体参数。 |

#### Action 参数

> `Priority` 默认 `Medium`；`SamePriorityOption` 默认 `ClearAndInterrupt`；TTS `AddHistory` 默认 `true`；LLM `AddQuestionToHistory` 默认 `false`，`AddAnswerToHistory` 默认 `true`。

<Tabs titleSize="h3">
<Tab title="SendAgentInstanceTTS">

| 参数 | 类型 | 是否必选 | 描述 |

| Text | String | 是 | 用于 TTS 的文本内容，最大不超过 300 个字符。 |

| SamePriorityOption | String | 否 | 相同优先级时的打断策略，默认为 `ClearAndInterrupt`。可选值：`ClearAndInterrupt`、`Enqueue`。 |
| InterruptMode | Number | 否 | 打断模式。 |
| EnqueueUserSpeech | Boolean | 否 | 是否排队用户语音。 |

请求示例：

#### 发送请求

客户端调用 Express `callExperimentalAPI` 向指定 `AgentUserId` 发送房间通道消息（msg_type=20）：

<CodeGroup>
```json (Android / iOS 原生端)
{
    "method": "liveroom.room.send_room_channel_message",
    "params": {
        "room_id": "room_001",
        "msg_type": 20,
        "msg_content": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"Text\":\"你好\",\"AddHistory\":true,\"Priority\":\"Medium\",\"SamePriorityOption\":\"ClearAndInterrupt\"}}",
        "user_list": ["<AgentUserId>"],
        "seq": 1
    }
}
```
```json (Web 端)
{
    "method": "sendRoomChannelMessage",
    "params": {
        "roomID": "room_001",
        "msgType": 20,
        "msgContent": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"Text\":\"你好\",\"AddHistory\":true,\"Priority\":\"Medium\",\"SamePriorityOption\":\"ClearAndInterrupt\"}}",
        "toUserIDList": ["<AgentUserId>"],
        "seq": 1
    }
}
```
</CodeGroup>

#### 接收响应

客户端在 `onRecvExperimentalAPI` 回调中接收响应（msg_type=22），响应消息体解码后格式如下：

```json
{
    "Action": "SendAgentInstanceTTS",
    "Seq": "client_A:web_01:1",
    "Code": 0,
    "Message": "success",
    "RequestId": "8916619579948490296",
    "Data": {
        "Round": 1481651956
    }
}
```

| 字段 | 类型 | 描述 |
| --- | --- | --- |
| Action | String | 原请求 Action。 |
| Seq | String | 原请求 Seq，客户端据此关联请求。 |
| Code | Int | 处理结果码，0 表示成功。 |
| Message | String | 处理结果说明，失败时包含错误信息。 |
| RequestId | String | 服务端返回的请求 ID。 |
| Data | Object | 响应数据，按不同 Action 返回不同内容。 |
| Data.Round | Int64 | 本轮 Round。 |

### 注意事项

#### AgentUserId 说明

数字人通话场景下，后端用于接收房间信令的目标 userID 与普通 1V1 智能体场景不同，客户端必须按以下规则决定目标 userID：

| 场景 | 目标 userID 取值 |
| --- | --- |
| 普通智能体（通过[创建智能体实例](/aiagent-server/api-reference/agent-instance-management/create-agent-instance)创建） | 接口响应中 `RTC.AgentUserId` 字段的值 |
| 数字人智能体（通过[创建数字人 Agent 实例](/aiagent-server/api-reference/agent-instance-management/create-digital-human-agent-instance)创建） | `ai_agent_<AgentInstanceId>`（AgentInstanceId 取自接口响应） |

#### Web 端与原生端协议差异

| 差异点 | Android / iOS 原生端 | Web 端 |
| --- | --- | --- |
| 发送方法名 | `liveroom.room.send_room_channel_message` | `sendRoomChannelMessage` |
| 接收回调方法名 | `liveroom.room.on_recive_room_channel_message` | `onRecvRoomChannelMessage` |
| 房间 ID 字段 | `room_id` | `roomID` |
| 接收方列表字段 | `user_list` | `toUserIDList` |

#### 链路追踪与重发去重

- `Seq` 由客户端按 `user_id:device_id:local_seq` 格式生成。
- 客户端按 `Seq` 维护请求表，对同一 `Seq` 的多个响应只处理第一个有效响应。
- 业务幂等由服务端处理，客户端无需关心。
