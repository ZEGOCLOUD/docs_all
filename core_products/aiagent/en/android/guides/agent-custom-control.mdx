# Agent Custom Control

Agent Custom Control allows the client to directly send agent instance control requests via **ZEGO Express SDK** room signaling, without relying on server-side API calls.

Through this feature, developers can implement the following control capabilities on the client side:

- **Trigger TTS**: Make the AI agent speak a segment of text via voice, suitable for fixed scripts (e.g., welcome messages, reminders).
- **Trigger LLM**: Send text to the AI agent, enabling it to reason and respond based on LLM, suitable for proactive speech scenarios requiring context.
- **Interrupt Agent**: Immediately stop the AI agent's current TTS/LLM process.
- **Control Listening**: In walkie-talkie mode, control the AI agent to start or stop listening to a specific user.

Typical use cases include:
- Playing fixed scripts such as welcome messages or timeout reminders.
- Proactively initiating topics during silences to improve interaction experience.
- Interrupting the agent at key moments and immediately broadcasting important content.

Applicable version: v2.13.0 and above.

## Prerequisites

You have completed the basic process as described in [Quick Start](./../quick-start.mdx).

## Implementation

### Approach 1: Using the Open-Source Component (Recommended)

Use the [ZEGO AI Agent Action Client SDK](https://github.com/ZEGOCLOUD/ai_agent_quick_start/tree/master/agentaction) to quickly integrate agent custom control capabilities.

This SDK supports Web, Android, iOS Swift, iOS Objective-C, and Flutter platforms, exposing the following unified methods:

- `sendAgentInstanceTTS`
- `sendAgentInstanceLLM`
- `interruptAgentInstance`
- `startListening`
- `stopListening`

#### Integration Steps

1. Copy the `agentaction/` directory for your target platform into your project.
2. Create a Client instance.
3. In the Sender, call Express `callExperimentalAPI` and pass the `formatedJson` generated by the SDK.
4. In the `onRecvExperimentalAPI` callback, forward the response to the SDK's `handleRoomChannelMessage` method.

### Approach 2: Custom Implementation

If you prefer not to use the open-source SDK, you can implement agent control manually. The principle is: the client sends room channel messages (msg_type=20) to the server via ZEGO Express SDK's `callExperimentalAPI`, and the server returns responses via room channel messages (msg_type=22).

#### API Call Examples

The following shows code examples for sending control requests and receiving responses on each platform using Express SDK. For complete runnable examples, see the [ai_agent_quick_start (agentaction)](https://github.com/ZEGOCLOUD/ai_agent_quick_start/tree/master/agentaction) repository.

##### Sending Requests

Send room channel messages (msg_type=20) using Express SDK's `callExperimentalAPI`:

<Tabs>
<Tab title="Android / iOS / Flutter">

The `callExperimentalAPI` call format is the same for native platforms and Flutter. Pass a JSON string as the request body:

```json
{
    "method": "liveroom.room.send_room_channel_message",
    "params": {
        "room_id": "room_001",
        "msg_type": 20,
        "msg_content": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"Text\":\"Hello\"}}",
        "user_list": ["<AgentUserId>"],
        "seq": 1
    }
}
```

The `msg_content` field is a JSON string containing:

| Field | Type | Description |
| --- | --- | --- |
| Action | String | Operation type, e.g. `SendAgentInstanceTTS`, `SendAgentInstanceLLM`, `InterruptAgentInstance`, `StartListening`, `StopListening` |
| Seq | String | Client-generated request sequence number, returned as-is in the response |
| Params | Object | Parameters corresponding to each Action |

> All platforms provide a unified SDK with 5 Action methods: `sendAgentInstanceTTS`, `sendAgentInstanceLLM`, `interruptAgentInstance`, `startListening`, `stopListening`. You can call these methods directly without manually constructing `msg_content`. For details, see the [agentaction SDK documentation](https://github.com/ZEGOCLOUD/ai_agent_quick_start/tree/master/agentaction).

</Tab>
<Tab title="Web">

The Web platform uses the `sendRoomChannelMessage` method with camelCase field names:

```json
{
    "method": "sendRoomChannelMessage",
    "params": {
        "roomID": "room_001",
        "msgType": 20,
        "msgContent": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"Text\":\"Hello\"}}",
        "toUserIDList": ["<AgentUserId>"],
        "seq": 1
    }
}
```

> **Note:** The Web platform does not have a send callback. Success is determined by whether `callExperimentalAPI` throws an error. Business results must wait for the agent response or timeout.

</Tab>
</Tabs>

##### Receiving Responses

Receive server responses (msg_type=22) via Express SDK callbacks:

<Tabs>
<Tab title="Android / iOS / Flutter">

On native and Flutter platforms, use the `onRecvExperimentalAPI` callback to receive room channel messages, then pass the content to `handleRoomChannelMessage` for parsing:

**Android (Java):**

```java
ZegoExpressEngine.createEngine(profile, appContext, new IZegoEventHandler() {
    @Override
    public void onRecvExperimentalAPI(String content) {
        client.handleRoomChannelMessage(content);
    }
});
```

**iOS (Swift):**

```swift
final class AgentActionExpressHandler: NSObject, ZegoExpressEventHandler {
    let actionClient: ZegoAIAgentActionClient

    init(actionClient: ZegoAIAgentActionClient) {
        self.actionClient = actionClient
    }

    func onRecvExperimentalAPI(_ content: String) {
        _ = actionClient.handleRoomChannelMessage(content: content)
    }
}
```

**Flutter:**

```dart
ZegoExpressEngine.onRecvExperimentalAPI = (content) {
    client.handleRoomChannelMessage(content);
};
```

</Tab>
<Tab title="Web">

On the Web platform, first enable the receiving channel via `callExperimentalAPI`, then receive responses via the `recvExperimentalAPI` event callback:

```js
// Enable receiving channel
zg.callExperimentalAPI({
    method: 'liveroom.room.on_recive_room_channel_message',
    params: {}
});

// Receive responses
zg.on('recvExperimentalAPI', (payload) => {
    // payload is the JSON object returned by the server
    console.log('recv:', JSON.stringify(payload));
    client.handleRoomChannelMessage(payload);
});
```

</Tab>
</Tabs>

#### Message Protocol

The client supports the following 5 control capabilities via room signaling:

| Action | Description | Corresponding Server API |
| --- | --- | --- |
| `SendAgentInstanceTTS` | Trigger TTS to make the AI agent speak a text segment. | [Trigger TTS](/aiagent-server/api-reference/agent-instance-control/send-agent-instance-tts) |
| `SendAgentInstanceLLM` | Trigger LLM to make the AI agent reason and respond based on text. | [Trigger LLM](/aiagent-server/api-reference/agent-instance-control/send-agent-instance-llm) |
| `InterruptAgentInstance` | Interrupt the agent, stopping the current TTS/LLM process. | [Interrupt Agent Instance](/aiagent-server/api-reference/agent-instance-control/interrupt-agent-instance) |
| `StartListening` | Make the agent start listening to a specific user. | [Start Listening](/aiagent-server/api-reference/agent-instance-control/start-listening) |
| `StopListening` | Make the agent stop listening to a specific user. | [Stop Listening](/aiagent-server/api-reference/agent-instance-control/stop-listening) |

**Message Types:**

| Message Type | Value | Direction | Description |
| --- | --- | --- | --- |
| Request | `20` | Client → Server | Client sends a control request. |
| Response | `22` | Server → Client | Server returns a processing response. |

**Message Body Structure (msg_content):**

```json
{
    "Action": "SendAgentInstanceTTS",
    "Seq": "client_A:web_01:1",
    "Params": {
        "Key": "Value"
    }
}
```

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| Action | String | Yes | Control action name, corresponding to the 5 Actions in the table above. |
| Seq | String | Yes | Business tracing identifier, generated by the client. Must be unique across multi-user/multi-device scenarios in the same room. Recommended format: `user_id:device_id:local_seq`. |
| Params | Object | Yes | Specific parameters for each Action. |

#### Action Parameters

> `Priority` defaults to `Medium`; `SamePriorityOption` defaults to `ClearAndInterrupt`; TTS `AddHistory` defaults to `true`; LLM `AddQuestionToHistory` defaults to `false`, `AddAnswerToHistory` defaults to `true`.

The following Tabs include parameter descriptions, request examples, sending request examples, and response formats for each Action.

The client receives responses (msg_type=22) in the `onRecvExperimentalAPI` callback:

```json
{
    "Action": "SendAgentInstanceTTS",
    "Seq": "client_A:web_01:1",
    "Code": 0,
    "Message": "success",
    "RequestId": "8916619579948490296",
    "Data": {
        "Round": 1481651956
    }
}
```

| Field | Type | Description |
| --- | --- | --- |
| Action | String | Original request Action. |
| Seq | String | Original request Seq, used by the client to correlate requests. |
| Code | Int | Processing result code, 0 indicates success. |
| Message | String | Processing result description, contains error info on failure. |
| RequestId | String | Request ID returned by the server. |
| Data | Object | Response data. |
| Data.Round | Int64 | Current round number. |

##### Sending Requests

The client calls Express `callExperimentalAPI` to send room channel messages (msg_type=20) to the specified `AgentUserId`:

<CodeGroup>
```json (Android / iOS Native)
{
    "method": "liveroom.room.send_room_channel_message",
    "params": {
        "room_id": "room_001",
        "msg_type": 20,
        "msg_content": "{\"Action\":\"StopListening\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"UserId\":\"user_123\"}}",
        "user_list": ["<AgentUserId>"],
        "seq": 1
    }
}
```
```json (Web)
{
    "method": "sendRoomChannelMessage",
    "params": {
        "roomID": "room_001",
        "msgType": 20,
        "msgContent": "{\"Action\":\"StopListening\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"UserId\":\"user_123\"}}",
        "toUserIDList": ["<AgentUserId>"],
        "seq": 1
    }
}
```
</CodeGroup>

##### Receiving Responses

The client receives responses (msg_type=22) in the `onRecvExperimentalAPI` callback:

```json
{
    "Action": "StopListening",
    "Seq": "client_A:web_01:1",
    "Code": 0,
    "Message": "success",
    "RequestId": "8916619579948490296",
    "Data": {
        "Round": 158422001
    }
}
```

| Field | Type | Description |
| --- | --- | --- |
| Action | String | Original request Action. |
| Seq | String | Original request Seq, used by the client to correlate requests. |
| Code | Int | Processing result code, 0 indicates success. |
| Message | String | Processing result description, contains error info on failure. |
| RequestId | String | Request ID returned by the server. |
| Data | Object | Response data. |
| Data.Round | Int64 | Current round number. |

</Tab>
</Tabs>

### Notes

#### AgentUserId

In digital human call scenarios, the target userID for receiving room signaling differs from regular 1V1 agent scenarios. The client must determine the target userID as follows:

| Scenario | Target userID Value |
| --- | --- |
| Regular agent (created via [Create Agent Instance](/aiagent-server/api-reference/agent-instance-management/create-agent-instance)) | `RTC.AgentUserId` field value from the API response |
| Digital human agent (created via [Create Digital Human Agent Instance](/aiagent-server/api-reference/agent-instance-management/create-digital-human-agent-instance)) | `ai_agent_<AgentInstanceId>` (AgentInstanceId from the API response) |

#### Web vs Native Protocol Differences

| Difference | Android / iOS Native | Web |
| --- | --- | --- |
| Send method | `liveroom.room.send_room_channel_message` | `sendRoomChannelMessage` |
| Receive callback | `liveroom.room.on_recive_room_channel_message` | `onRecvRoomChannelMessage` |
| Room ID field | `room_id` | `roomID` |
| Recipient list field | `user_list` | `toUserIDList` |

#### Tracing and Deduplication

- `Seq` is generated by the client in the format `user_id:device_id:local_seq`.
- The client maintains a request table by `Seq` and only processes the first valid response for the same `Seq`.
- Business idempotency is handled by the server; the client does not need to handle it.
