# Agent Custom Control

Agent custom control allows the client to directly initiate agent instance control requests through the **ZEGO Express SDK** room signaling, without relying on server API calls.

With this feature, developers can implement the following control capabilities on the client side:

- **Proactively call TTS**: Have the agent read a piece of text aloud using voice, suitable for broadcasting fixed scripts (such as welcome messages, reminders, and so on).
- **Proactively call LLM**: Give the agent a piece of text and have it infer and reply based on the LLM, suitable for proactive speaking scenarios that require context.
- **Interrupt the agent**: Immediately interrupt the TTS/LLM flow currently in progress on the agent.
- **Control listening**: In walkie-talkie mode, control the agent to start or stop listening to a specified user.

Typical use scenarios include:
- Broadcasting fixed scripts such as welcome messages and timeout reminders.
- Proactively initiating a topic when the conversation stalls to improve the interaction experience.
- Interrupting the agent and immediately broadcasting important content when a key node is triggered.

Applicable versions: v2.13.0 and above.

## Prerequisites

You should have already completed the basic flow by referring to [Quick Start](./../quick-start.mdx).

## Usage

### Method 1: Implement via the open-source component (Recommended)

Using the [ZEGO AI Agent Action client suite](https://github.com/ZEGOCLOUD/ai_agent_quick_start/tree/master/agentaction), you can quickly integrate the agent custom control capability.

This suite supports five platforms — Web, Android, iOS Swift, iOS Objective-C, and Flutter — and exposes the following unified methods:

- `sendAgentInstanceTTS`
- `sendAgentInstanceLLM`
- `interruptAgentInstance`
- `startListening`
- `stopListening`

#### Integration steps

1. Copy the `agentaction/` directory for the target platform into your own project.
2. Create a Client instance.
3. In the Sender, call the Express `callExperimentalAPI` and pass through the `formatedJson` generated by the suite.
4. In the `onRecvExperimentalAPI` callback, forward the response content to the suite's `handleRoomChannelMessage` method.

The constructor parameters for each platform are described below. Because Android / OC do not support default parameters, all parameters must be passed explicitly (in a standard agent scenario, the digital-human-related parameters can be passed as empty values).

|Parameter name-150px | Description | Example value|
| --- | --- | --- |
| roomId | The business room ID, which is the same as the ZEGOCLOUD audio/video room ID. | `room_001` |
| agentUserId | The target agent userID. For a standard agent, take `rtcInfo.agentUserId`; for digital-human scenarios, it is automatically calculated by the suite as `ai_agent_<agentInstanceId>`, and this parameter is ignored. | `<AgentUserId>` |
| userId | The current end-user ID, used to generate the local tracing Seq. | `client_A` |
| agentInstanceId | The agent instance ID. Required for digital-human scenarios; for standard scenarios, it is required on Android / OC (pass `null` / `nil`), can be omitted on Swift / Flutter (defaults to nil), and does not exist on Web. | `null` (pass the instance ID for digital humans) |
| isDigitalHuman | Whether it is a digital-human call, which determines whether the `ai_agent_<instanceId>` concatenation rule is used. For standard scenarios, pass `false` / `NO` or omit it. | `false` (pass `true` for digital humans) |
| deviceId | The device ID, used to construct the Seq; if empty, it is automatically generated by the suite. | `null` (auto-generated if empty) |
| timeoutMs | The default timeout for a single request (ms), default 5000. | `5000` |
| sender | The sending callback implemented by the business, which passes the `formatedJson` assembled by the suite through to the Express `callExperimentalAPI`. | No fixed value, see the sender implementation of each platform above |
| onResponse | The global response callback, triggered each time a PaaS response is received (optional). | No fixed value, see the onResponse implementation of each platform above |
| onError | The global error callback, triggered on send failure / timeout / proactive cancellation (optional). | No fixed value, see the onError implementation of each platform above |

### Method 2: Custom implementation

If you do not want to use the open-source suite, you can also implement the agent control yourself. The principle is: the client sends a room channel message (msg_type=20) to the server through the ZEGO Express SDK's `callExperimentalAPI`, the server processes it and returns a response through a room channel message (msg_type=22), and the client receives the response through `onRecvExperimentalAPI`.

#### Call the callExperimentalAPI interface to send commands

The following shows how to use the ZEGO Express SDK on each platform to send control commands. For complete, runnable example code, refer to [ai_agent_quick_start (agentaction)](https://github.com/ZEGOCLOUD/ai_agent_quick_start/tree/master/agentaction).

Use the Express SDK's `callExperimentalAPI` to send a room channel message (msg_type=20). The calling method for each platform is as follows:

<Tabs>
<Tab title="Android / iOS / Flutter">

The `callExperimentalAPI` calling method on Android, iOS, and Flutter is the same, with the `method` parameter using `liveroom.room.send_room_channel_message`. Pass in the request body in JSON string format:

```json
{
    "method": "liveroom.room.send_room_channel_message",
    "params": {
        "room_id": "room_001",
        "msg_type": 20,
        "msg_content": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"Text\":\"Hello\"}}",
        "user_list": ["<AgentUserId>"],
        "seq": 1
    }
}
```
</Tab>
<Tab title="Web">

On Web, the `method` parameter uses `sendRoomChannelMessage`, and the field naming uses camelCase. Pass in the request body in JSON string format:

<Warning title="Note"> Sending on Web has no callback; whether the send succeeds is determined by whether `callExperimentalAPI` throws an error, and the business result must wait for the agent to respond or time out.</Warning>

```json
{
    "method": "sendRoomChannelMessage",
    "params": {
        "roomID": "room_001",
        "msgType": 20,
        "msgContent": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Params\":{\"Text\":\"Hello\"}}",
        "toUserIDList": ["<agent_user_id>"],
        "seq": 1
    }
}
```
</Tab>
</Tabs>

##### params.user_list parameter

The `user_list` on Android, iOS, and Flutter, and the `toUserIDList` on Web, are both the target agent userID (pass only one).

<Warning title="Note">
In a digital-human call scenario, the target agent userID passed to user_list (toUserIDList) is different from that in a standard 1-on-1 agent scenario. The client must pass it according to the following rules:

| Scenario | Target agent userID value |
| --- | --- |
| Standard agent (created via [Create Agent Instance](/aiagent-server/api-reference/agent-instance-management/create-agent-instance)) | The same as the value of the `RTC.AgentUserId` field in the interface request parameters. For example, `agent_123456` |
| Digital-human agent (created via [Create Digital Human Agent Instance](/aiagent-server/api-reference/agent-instance-management/create-digital-human-agent-instance)) | Generated according to the `ai_agent_<AgentInstanceId>` template. AgentInstanceId is the Data.AgentInstanceId in the response parameters. For example, `ai_agent_1912122918452641792` |


</Warning>

##### params.msg_content parameter

Here `msg_content` is a JSON string containing the following fields:

|Field-100px | Type | Description|
| --- | --- | --- |
| Action | String | The operation type, such as `SendAgentInstanceTTS`, `SendAgentInstanceLLM`, `InterruptAgentInstance`, `StartListening`, `StopListening`. See [Action](#action) |
| Seq | String | The business tracing identifier, generated by the client, and must be unique across multi-user/multi-device scenarios in the same room. The recommended format is `user_id:device_id:local_seq`. It will be returned as-is in onRecvExperimentalAPI.|
| Params | Object | The parameters corresponding to each Action. See [Params](#params) |


###### Action

In the msg_content parameter of the callExperimentalAPI interface, the Params field of the JSON string is as follows. Through room signaling, the client supports the following 5 categories of control capabilities:

| Action | Capability description | Corresponding server interface |
| --- | --- | --- |
| `SendAgentInstanceTTS` | Proactively call TTS to have the agent read a piece of text aloud. | [Proactively call TTS](/aiagent-server/api-reference/agent-instance-control/send-agent-instance-tts) |
| `SendAgentInstanceLLM` | Proactively call LLM to have the agent infer based on text and reply. | [Proactively call LLM](/aiagent-server/api-reference/agent-instance-control/send-agent-instance-llm) |
| `InterruptAgentInstance` | Interrupt the agent and stop the current TTS/LLM flow. | [Interrupt agent instance](/aiagent-server/api-reference/agent-instance-control/interrupt-agent-instance) |
| `StartListening` | The agent starts listening to a specified user. | [Start agent listening](/aiagent-server/api-reference/agent-instance-control/start-listening) |
| `StopListening` | The agent stops listening to a specified user. | [Stop agent listening](/aiagent-server/api-reference/agent-instance-control/stop-listening) |

#### Listen for onRecvExperimentalAPI to receive the response

After the server finishes processing, it delivers the response through a room channel message (msg_type=22), and the client receives it in the `onRecvExperimentalAPI` callback of the ZEGO Express SDK. The callback content is a JSON string (native) or an object (Web), where `msg_content`/`msgContent` is itself another layer of JSON string and needs to be **parsed twice** to obtain the response body. Finally, the response is associated back to the original request through the `Seq` in the response body.

<Tabs>
<Tab title="Android / iOS / Flutter">

On native platforms, implement the `onRecvExperimentalAPI(String content)` callback when creating the `ZegoExpressEngine`. `content` is a JSON string (fields use snake_case), and its structure is the same as when sending:

```json
{
    "method": "liveroom.room.on_recive_room_channel_message",
    "params": {
        "room_id": "room_001",
        "msg_type": 22,
        "msg_content": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Code\":0,\"Message\":\"success\",\"RequestId\":\"8916619579948490296\",\"Data\":{\"Round\":1481651956}}"
    }
}
```

Taking Android (Java) as an example, perform a second parse on `params.msg_content` to extract the response body:

```java
ZegoExpressEngine.createEngine(profile, appContext, new IZegoEventHandler() {
    @Override
    public void onRecvExperimentalAPI(String content) {
        try {
            JSONObject data = new JSONObject(content);
            // Only process business response callbacks (msg_type=22)
            if (!"liveroom.room.on_recive_room_channel_message".equals(data.optString("method"))) {
                return;
            }
            JSONObject params = data.optJSONObject("params");
            if (params == null || params.optInt("msg_type", -1) != 22) {
                return;
            }
            // msg_content is a nested JSON string that needs to be parsed twice to obtain the response body
            JSONObject resp = new JSONObject(params.optString("msg_content"));
            String seq = resp.optString("Seq");
            int code = resp.optInt("Code");
            // Associate back to the original request by seq; code == 0 indicates success, others are business error codes
        } catch (JSONException e) {
            e.printStackTrace();
        }
    }
});
```

The parsing logic for iOS (Objective-C / Swift) and Flutter is the same: first check that `method` is `liveroom.room.on_recive_room_channel_message`, then perform a second parse on `params.msg_content` to extract the response body, and associate the request by `Seq`.

</Tab>
<Tab title="Web">

On Web, the response is received through the `zg.on('recvExperimentalAPI', payload => {...})` event, and `payload` is an object (fields use camelCase). `payload.msgContent` is also a nested JSON string that needs to be parsed twice. Some Web Express versions need to explicitly enable room channel message reception first:

```json
{
    "method": "onRecvRoomChannelMessage",
    "content": {
        "roomID": "room_001",
        "msgType": 22,
        "msgContent": "{\"Action\":\"SendAgentInstanceTTS\",\"Seq\":\"client_A:web_01:1\",\"Code\":0,\"Message\":\"success\",\"RequestId\":\"8916619579948490296\",\"Data\":{\"Round\":1481651956}}"
    }
}
```

```javascript
// Some Web Express versions need to explicitly enable room channel message reception
zg.callExperimentalAPI({ method: 'onRecvRoomChannelMessage', params: {} });

// Listen for the experimental API callback and parse the response
zg.on('recvExperimentalAPI', (payload) => {
    if (payload.method !== 'onRecvRoomChannelMessage') {
        return;
    }
    const body = payload.content || payload.params;
    if (body.msgType !== 22) {
        return;
    }
    // msgContent is a nested JSON string that needs to be parsed twice to obtain the response body
    const resp = JSON.parse(body.msgContent);
    const { Seq, Code, Message } = resp;
    // Associate back to the original request by Seq; Code === 0 indicates success, others are business error codes
});
```

### Notes

#### Protocol differences between Web and native

| Difference | Android / iOS native | Web |
| --- | --- | --- |
| Send method name | `liveroom.room.send_room_channel_message` | `sendRoomChannelMessage` |
| Receive callback method name | `liveroom.room.on_recive_room_channel_message` | `onRecvRoomChannelMessage` |
| Room ID field | `room_id` | `roomID` |
| Recipient list field | `user_list` | `toUserIDList` |

#### Tracing and retransmission deduplication

- `Seq` is generated by the client in the `user_id:device_id:local_seq` format.
- The client maintains a request table by `Seq`, and for multiple responses with the same `Seq`, only the first valid response is processed.
- Business idempotency is handled by the server, and the client does not need to worry about it.
