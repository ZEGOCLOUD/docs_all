# Using Token Authentication

---

## Feature Overview

Authentication refers to verifying whether a user has permission to access the system, to avoid security risks caused by missing permission controls or improper operations. ZEGOCLOUD authenticates users through Tokens (including basic authentication Tokens and privilege authentication Tokens).

A basic authentication Token means that the developer must pass the Token to the SDK via the [setCustomToken](https://doc-zh.zego.im/API/ServerRecord/namespace_z_e_g_o_1_1_l_i_v_e_r_o_o_m.html#a68bdea0d66d62c8bf4d671284707b3ae) API before logging in to a room, to verify the user's legitimacy.

A privilege authentication Token refers to further improving security by enabling the room ID privilege field, which can verify the room ID for login.


## Prerequisites

- Basic authentication Token is enabled by default. For privilege authentication Token (i.e., room ID privilege field), please contact ZEGOCLOUD Technical Support to enable it.
- You have integrated the ZEGOCLOUD Local Server Recording SDK in your project. See [Quick Start - Integration](./SDK%20Integration.mdx).

## Implementation Flow

When using Token authentication, the developer needs to generate a Token first, and the ZEGOCLOUD server validates the user carrying the Token.

The following example demonstrates the flow using Token to determine whether a user can log in to a room:

<Frame width="512" height="auto" caption="">
  <img src="https://doc-media.zego.im/sdk-doc/Pics/QuickStart/token_uml_liveroom.png" />
</Frame>

1. The developer's client sends a Token request.
2. The Token is generated on the developer's server and returned to the developer's client.
3. The developer's client passes the obtained Token and roomID information to the SDK via the [setCustomToken](https://doc-zh.zego.im/API/ServerRecord/namespace_z_e_g_o_1_1_l_i_v_e_r_o_o_m.html#a68bdea0d66d62c8bf4d671284707b3ae) API.
4. The ZEGOCLOUD SDK automatically sends the Token to the ZEGOCLOUD server for validation.
5. The ZEGOCLOUD server returns the validation result to the ZEGOCLOUD SDK.
6. The ZEGOCLOUD SDK returns the validation result directly to the developer's client. Unauthorized clients will fail to log in.

## Steps

This section describes in detail how developers can generate and use Tokens.

###  Obtain AppID and ServerSecret

Generating a Token requires the developer's project unique identifier AppID and secret key ServerSecret. Obtain them as described in "Project Information" under [Console - Project Management](/console/project-info).


###  Generate Token

#### Server-Side Token Generation (Recommended)

After obtaining the project's AppID and ServerSecret, developers can generate Tokens on their own server based on business requirements. The developer's client sends a Token request to the developer's server, which generates the Token and returns it to the corresponding client.

ZEGO provides an open-source zego_server_assistant plugin on GitHub/Gitee, supporting Go, C++, Java, Objective-C, Python, PHP, .NET, and Node.js for Token generation on the developer's server or client.

The zego_server_assistant plugin provides sample code for both basic authentication Token and privilege authentication Token. Their applicable scenarios are as follows; developers can choose as needed:
- Basic authentication Token: Used for simple business permission validation scenarios. The "payload" field can be left empty.
- Privilege authentication Token: For further improved security, the room ID privilege field is enabled to verify the room ID for login. The "payload" field must be generated according to rules. See the "Privilege Authentication Token" sample code in the table below.

<Warning title="Note">

- For security, it is strongly recommended to generate Tokens on the server side; otherwise, there is a risk of ServerSecret being compromised.
- Please use "token04" in zego_server_assistant to generate Tokens.
</Warning>

Using Go as an example, developers can follow these steps to use zego_server_assistant to generate a Token:

1. Use the command `go get github.com/zegoim/zego_server_assistant` to get the dependency package.
2. In your code, import the plugin via `import "github.com/zegoim/zego_server_assistant/token/go/src/token04"`.
3. Call the GenerateToken04 method provided by the plugin to generate a Token.

Tokens can be generated in the following two ways.

- **Method 1: Basic Authentication Token**

  Sample code for generating a basic authentication Token:

```go
package main

import (
    "fmt"
    "github.com/zegoim/zego_server_assistant/token/go/src/token04"
)

/*
Sample code for basic authentication Token generation
*/


func main() {
    var appId uint32 = 1                               // Numeric ID assigned by ZEGO, unique identifier for each developer
    userId := "demo"                                   // User ID
    serverSecret := "fa94dd0f974cf2e293728a1234567890" // AES encryption key used when obtaining the token
    var effectiveTimeInSeconds int64 = 3600            // Token validity duration in seconds
    var payload string = ""                            // Token business authentication extension; leave empty for basic authentication

    // Generate token
    token, err := token04.GenerateToken04(appId, userId, serverSecret, effectiveTimeInSeconds, payload)
    if err != nil {
	fmt.Println(err)
	return
    }
    fmt.Println(token)
}
```

- **Method 2: Privilege Authentication Token**

  Sample code for generating a privilege authentication Token:

<Warning title="Note">


The "RoomId" must be provided when generating a privilege authentication Token.
</Warning>

```go
package main

import (
    "encoding/json"
    "fmt"
    "github.com/zegoim/zego_server_assistant/token/go/src/token04"
)

/*
Sample code for privilege authentication Token generation
*/

// Token business extension: Privilege authentication properties
type RtcRoomPayLoad struct {
    RoomId       string      `json:"room_id"`        // Room ID (required); used for strong verification of the room ID
    Privilege    map[int]int `json:"privilege"`      // Privilege field toggle list; used for strong verification of API operation permissions
    StreamIdList []string    `json:"stream_id_list"` // Stream list; used for strong verification of stream IDs; can be empty; if empty, stream IDs are not verified
}

func main() {
    var appId uint32 = 1                               // Numeric ID assigned by ZEGO, unique identifier for each developer
    roomId := "demo"                                   // Room ID
    userId := "demo"                                   // User ID
    serverSecret := "fa94dd0f974cf2e293728a526b028271" // AES encryption key used when obtaining the token
    var effectiveTimeInSeconds int64 = 3600            // Token validity duration in seconds

    // Refer to github.com/zegoim/zego_server_assistant/token/go/src/token04/token04.go for definitions
    // Privilege field definitions
    //const (
    //	PrivilegeKeyLogin   = 1 // Login privilege authentication
    //	PrivilegeKeyPublish = 2 // Publish stream privilege authentication
    //)

    // Privilege toggle definitions
    //const (
    //	PrivilegeEnable     = 1 // Permission granted
    //	PrivilegeDisable    = 0 // No permission
    //)

    // Business privilege authentication configuration; multiple privilege fields can be configured
    privilege := make(map[int]int)
    privilege[token04.PrivilegeKeyLogin] = token04.PrivilegeEnable    // Has room login permission
    privilege[token04.PrivilegeKeyPublish] = token04.PrivilegeDisable // No publish stream permission

    // Token business extension configuration
    payloadData := &RtcRoomPayLoad{
	RoomId:       roomId,
	Privilege:    privilege,
	StreamIdList: nil,
    }

    payload, err := json.Marshal(payloadData)
    if err != nil {
	fmt.Println(err)
	return
    }

    // Generate token
    token, err := token04.GenerateToken04(appId, userId, serverSecret, effectiveTimeInSeconds, string(payload))
    if err != nil {
	fmt.Println(err)
	return
}
    fmt.Println(token)
}

```



<Warning title="Note">When running the Java source code for Token generation, if you encounter the "java.security.InvalidKeyException: illegal Key Size" exception, please refer to the [related FAQ documentation](https://doc-zh.zego.im/faq/java_illegal_key_size_solution?product=ExpressVideo&platform=all) for a solution.</Warning>


#### Client-Side Token Generation (Not Recommended)

If you cannot distribute Tokens from the server during development, you can temporarily generate Tokens using client-side code, and complete the integration with the server later.


<Warning title="Note">

- When publishing your app, never generate Tokens on the client side; otherwise, your ServerSecret will be at risk of exposure.
- For security, it is strongly recommended to generate Tokens on the server side; otherwise, there is a risk of ServerSecret being compromised.
</Warning>
