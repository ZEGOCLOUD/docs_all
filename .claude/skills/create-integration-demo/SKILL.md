# 接入测试 — 构建验证

模拟零经验开发者按文档操作，实际构建项目、编写代码、编译验证，发现真实问题。属于接入测试的核心执行阶段。

## 核心人设

**零经验开发者**：
- 没有任何 ZEGO 产品经验（RTC、IM、AI Agent 等均未使用过）
- 不熟悉实时通讯行业概念（房间、Token、鉴权、推流等）
- 依赖文档解释一切术语和概念
- 不会猜测行业惯例，不会主动翻源码
- 有比较强的 UI 审美。涉及写 UI 时都能写出一个比较美观、现代的 UI。（不过不会为了美观而加一堆skill，更多是通过布局、尺寸来使整体美观）

**严格文档执行者**：
- 只做文档描述的事情，不补充文档未提及的步骤
- 如果文档没说要做某件事，就不做

**示例代码使用原则**：
- 文档明确提到"可参考示例代码"或"下载示例代码"时 → 可以查看和参考示例代码，这属于正常指引
- 文档没有提到示例代码，但文档指引已经无法让你继续编写代码时 → 才能去查看示例代码作为 fallback，但必须记录为 📄 文档问题
- 凡是依赖示例代码来补足文档缺失的步骤、流程、前置条件的，一律记录为 📄 文档问题
- 集成项目代码必须依据文档从头编写，不能直接下载官方示例代码修改

## 输入参数

通过 AskUserQuestion 或上下文获取：

- **文档路径** — 要验证的 MDX 文档文件路径
- **产品名称** — 产品名（如 AIAgent、实时音视频、即时通讯）
- **输出目录** — 项目代码和测试文件存放位置，格式 `examples/{产品名}-{平台}-access-test/`

## 完整流程

### Step 1: 识别目标平台

读取目标文档，从以下线索识别平台：

1. **标题和章节名** — "Web 端集成"、"Android 快速开始"
2. **代码示例语言** — JS/TS → Web, Kotlin/Java → Android, Swift → iOS, Dart → Flutter
3. **框架关键词** — React、Vue、Flutter、uni-app、Electron
4. **依赖和工具链** — npm → Web, Gradle → Android, CocoaPods → iOS, pub → Flutter

**平台映射**：

| 识别结果 | 构建目标 | 自动化测试 |
|---------|---------|-----------|
| Web (React/Vue/原生) | Web 应用 | @zegocloud/auto-web |
| Android 原生 | Android APK | @midscene/android |
| iOS 原生 | iOS 应用 | @midscene/ios |
| Flutter | Android APK | @midscene/android |
| React Native | Android APK | @midscene/android |
| uni-app | Android APK | @midscene/android |
| Electron | macOS 应用 | @midscene/computer |

### Step 2: 模拟零经验开发者阅读路径

按以下顺序逐步走查文档，**每步记录状态**：

1. **前提条件检查** — 文档要求准备什么？能否理解？
2. **环境准备** — 安装什么工具？命令能直接复制执行吗？
3. **项目创建** — 文档告诉怎么创建项目了吗？
4. **SDK 集成** — 依赖怎么装？初始化代码在哪？
5. **代码实现** — 按文档的代码示例写代码
6. **功能验证** — 文档告诉怎么验证功能是否正常吗？

**跟随文档内所有链接**：文档中引用的其他文档链接视为文档指引的一部分。按以下方式处理：

| 链接格式 | 处理方式 |
|----------|---------|
| @ 短链 | 用 `api-short-link-to-mdx` skill 解析到 API MDX 文件，读取 API 说明 |
| 相对路径 | 基于当前文件路径解析读取 |
| URL path | 用 `url-to-mdx` skill 解析到本地 MDX 文件并读取 |
| 完整 URL | 用 `url-to-mdx` 尝试解析；无法解析则用 Web 工具读取 |
| 页内锚点 | 在当前文档中查找对应标题 |

如果链接指向的文档确实提供了所需信息，则不构成"缺失"问题。如果需要跳转超过 2 层才能找到关键操作信息，可标记为 🟡 体验问题（信息过于分散）。

**状态标记**：
- ✅ 成功：文档说明清晰，步骤可直接执行
- ❌ 失败：文档缺失关键信息、步骤有误、代码跑不通
- ⚠️ 部分成功：需要猜测或外部搜索才能继续

**每条记录必须包含**：文档位置（章节名/行号）+ 引用原文 + 实际操作 + 结果

### Step 3: 收集运行所需的凭证

**从文档前提条件中提取**所有需要用户提供的信息：
- ZEGO 控制台 AppID / AppSign / AppSecret
- 需要开通的服务
- 服务器地址 / API 密钥
- Token 生成所需密钥
- 其他第三方服务凭证

使用 AskUserQuestion 向用户收集。

**前提条件充分性检查**：
- ✅ 充分：收集到的信息足以填入所有代码占位符 → 继续
- ❌ 不充分：代码中存在文档前提条件未提及的凭证 → 记录为 🔴 阻塞问题（📄 文档问题）

### Step 4: 构建项目

在输出目录下按文档指引创建项目：
- 使用文档中的命令（如 `npm create vite`、`flutter create`）
- 使用文档中指定的依赖版本号
- 按文档中的目录结构组织代码
- 如果文档没有说明如何创建项目，尝试推断最常见的方式

### Step 5: 编写代码

**按文档示例从头编写代码**，不能直接下载官方示例代码修改：
- 复制文档中的代码块到对应文件
- 替换占位符为测试值（如 `yourAppID` → 环境变量或实际值）
- 如果文档代码中有 `...` 省略部分，基于上下文合理补全
- **严格按文档内容编写，不添加文档未提及的逻辑**
- **代码必须从头编写，禁止下载官方示例代码后修改**

**猜测必须标注**：当代码中包含文档未明确说明的内容（如加密算法选择、参数格式推断、API 行为假设），必须用注释明确标注：

```javascript
// ⚠️ 推断: 文档未说明 token04 的具体加密方式，此处基于 [推断依据] 使用 HMAC-SHA256
```

推断依据必须具体（如"ZEGO 文档中其他产品使用该方式"、"npm 包 README 提到"、"参考链接指向的 API 文档"），不能为空。如果完全无法推断，标注为"文档未说明，无法推断，此处留空/使用默认值"。

**禁止编造后归咎文档**：不能把自己从外部知识引入的技术细节当作事实，然后说"文档说的不对"。如果文档没提到某个技术细节，那只是"未提及"，不是"错误"。

**示例代码使用规则**（同核心人设中的原则）：
- 文档说"可参考示例代码" → 可以看，正常操作
- 文档没提示例代码，但写不下去 → 可以看，但记为 📄 文档问题
- 任何时候依赖示例代码补足文档缺失 → 记为 📄 文档问题

**每段代码记录状态**：
- **能直接运行** — 复制粘贴即可
- **需要修改才能运行** — 记录需要修改什么
- **缺少关键信息无法运行** — 记录缺少什么

### Step 6: 编译验证

尝试编译/运行项目：

| 平台 | 编译命令 |
|------|---------|
| Web | `npm install && npm run build` |
| Android | `./gradlew assembleDebug` |
| iOS | `xcodebuild -workspace *.xcworkspace -scheme * build` |
| Flutter | `flutter build apk` |
| Node.js | `npm install && npm start` |

编译出错时：
1. 记录完整错误信息
2. 按决策树归类问题原因（详见 `references/problem-attribution.md`）
3. 判断非文档指引问题则尝试修复，记录修复步骤
4. 无法修复则记录为阻塞问题

### Step 7: 记录所有问题

将整个流程中发现的所有问题整理为结构化记录。

**问题归类**（详见 `references/problem-attribution.md`）：

| 类别 | 标记 | 定义 |
|------|------|------|
| 文档问题 | 📄 | 文档缺失、错误、模糊导致的问题 |
| 代码实现问题 | 🔧 | 按文档实现但代码仍有问题 |
| 环境问题 | 🏗️ | 环境配置、依赖版本、工具链问题 |
| 链接问题 | 🔗 | 文档中的外部链接失效 |
| 概念缺失 | 📖 | 文档未解释关键概念 |

**每个问题记录**：
1. 所在步骤
2. 文档原文引用（章节名 + 内容）
3. 实际操作和结果
4. 问题归类（按决策树判断）
5. 严重度（🔴 阻塞 / 🟡 体验差 / 🟢 优化建议）
6. **信息来源**（必须标注）：
   - **文档原文**：引用文档实际写的内容
   - **推断（附依据）**：文档未说明，基于某某依据推断

**问题归类必须区分信息来源**：

| 来源 | 归类规则 |
|------|---------|
| 文档明确写了 X，但 X 有误 | 📄 文档问题（错误） |
| 文档应该写但没写 X（已确认文档中确实没有链接指向 X） | 📄 文档问题（缺失） |
| 文档没说，推断的 Y 不对 | 🔧 代码实现问题，标注推断依据，不能归咎于文档 |
| 文档没说，猜的 Z 导致了问题 | 🔧 代码实现问题，标注"推断依据：..." |
