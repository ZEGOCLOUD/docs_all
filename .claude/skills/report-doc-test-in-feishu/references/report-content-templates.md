## 校正后文档评测汇总报告

写入本地：

```text
doc-test-reports/{run-name}/doc-eval-corrected/doc-eval-summary.md
```

模板沿用 `doc-eval` 的汇总报告结构。写入飞书前，`scripts/prepare-feishu-report-content.js` 会将本地 H1 对齐为 manifest 中的飞书文档标题，例如：

```markdown
# 【文档评测报告】汇总报告
```

`角色评分` 表格的 `报告` 列在写飞书前必须使用飞书链接，不要保留本地相对路径。创建飞书骨架后，运行 `scripts/prepare-feishu-report-content.js` 从 `feishu-report-manifest.json` 读取各角色文档 URL 并替换。

`角色评分` 表格后必须追加：

```markdown
**请阅读各角色详细报告文档并评论问题解决方案！**
```

同时增加：

```markdown
## 集成验证校正

| 原问题 | 原判断 | 构建/测试验证结果 | 校正后判断 | 说明 |
|--------|--------|------------------|------------|------|
| [DOC-001](../doc-eval/role-client-dev.md#doc-001) | 🔴 阻塞 | 构建复现 | 🔴 阻塞 | {说明} |
| [DOC-002](../doc-eval/role-support.md#doc-002) | 🔴 阻塞 | 未阻塞构建/测试 | 🟡 体验差 | {说明} |

## 集成验证新增文档问题

| 问题 ID | 来源 | 严重度 | 文档定位 | 问题 | 建议 |
|---------|------|--------|----------|------|------|
| DOC-N-001 | BUILD-003 | 🔴 阻塞 | {GitHub 链接} | {问题} | {建议} |
```

## 校正后角色报告

写入本地：

```text
doc-test-reports/{run-name}/doc-eval-corrected/role-*.md
```

以原始 `doc-eval/role-*.md` 为底稿生成校正版角色报告。不要只在末尾追加校正结论；必须把构建验证和自动化测试结果融合回原报告正文，让读者在问题摘要和问题详情里直接看到校正后的判断。

校正方式：

1. 保留原角色报告的主体结构和原问题 ID。
2. `阶段结论` 中的综合评分、阻塞问题数、体验问题数、优化建议数和一句话结论，要按构建/测试证据重新计算或说明保持不变的原因。
3. 原始角色报告中已有的 `定位` GitHub 链接必须原样保留，不能改写成文件名、章节名、所有文档或跨角色引用。
4. `问题摘要` 表格增加一列 `构建测试校正后严重程度`。
5. `问题详情` 中保留原始严重度，同时增加构建/测试校正结论。
6. `为什么是问题` 和 `建议` 要根据构建/测试结果适当改写，不能只保留原始纸面评测判断。
7. 构建/测试证据证明不成立的问题，不删除，标记为降级或撤销，并解释原因。
8. 构建/测试新增且属于文档质量的问题，追加为新的 `DOC-N-xxx` 问题，并在关联问题中指向 BUILD/TEST 来源，同时补充新的 GitHub 定位。

校正版角色报告中的关键字段：

```markdown
## 问题摘要

| 问题 ID | 类别 | 原严重度 | 构建测试校正后严重程度 | 问题 | 建议 |
|---------|------|----------|--------------------------|------|------|
| DOC-001 | 📄 文档问题 | 🔴 阻塞 | 🟡 体验差 | {校正后的一句话问题} | {校正后的一句话建议} |

## 问题详情

### DOC-001: {问题标题}

- 定位：[docs/xxx.mdx:123](https://github.com/ZEGOCLOUD/docs_all/blob/main/docs/xxx.mdx?plain=1#L123)
- 章节路径：{可选}
- 类别：📄 文档问题 / 🔗 链接问题 / 📖 概念缺失
- 原严重度：🔴 阻塞 / 🟡 体验差 / 🟢 优化建议
- 构建测试校正后严重程度：🔴 阻塞 / 🟡 体验差 / 🟢 优化建议 / 已撤销
- 构建/测试证据：[BUILD-001](../build-verification/build-issues.md#build-001) / [TEST-001](../auto-test/test-report.md#test-001) / 未复现
- 关联问题：{关联的 BUILD/TEST/DOC 问题；无则写无}

**原文内容**

```markdown
{摘录文档原文，包含必要上下文。}
```

**实际阅读结果**

{保留或改写原角色视角下的阅读结果。}

**构建/测试校正结果**

{说明构建验证和自动化测试是否复现该问题、是否改变严重度、是否新增证据。}

**为什么是问题**

{结合角色阅读影响和构建/测试结果重新说明。比如：构建阶段已阻塞，所以保持🔴；自动化测试未阻塞但造成额外查找成本，所以降为🟡；证据表明原判断不成立，所以撤销。}

**建议**

{根据构建/测试后的真实影响给出建议。优先给出应补充到文档的内容、位置、示例代码或告警说明。}
```

可在报告末尾保留 `## 集成验证校正记录` 作为索引，但不能用它替代上面对问题摘要和问题详情的正文校正。

| 最终接入判断 | 可接入 / 有条件可接入 / 不可接入 |

一句话结论：{整体评估。}
