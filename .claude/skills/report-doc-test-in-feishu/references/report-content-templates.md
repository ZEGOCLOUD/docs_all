# 报告内容模板

## 构建与自动化测试报告

写入本地：

```text
doc-test-reports/{run-name}/build-test-report.md
```

模板：

```markdown
# 构建与自动化测试报告

## 基本信息

| 字段 | 内容 |
|------|------|
| run-name | {run-name} |
| 产品 | {product} |
| 平台 | {platform} |
| 范围 | {scope} |
| 运行日期 | {date} |
| 示例项目 | doc-test-reports/{run-name}/examples/{demo-name}/ |

## 构建验证结论

| 项目 | 结论 |
|------|------|
| 是否成功生成项目 | 是 / 否 |
| 是否安装依赖成功 | 是 / 否 |
| 是否编译通过 | 是 / 否 |
| 是否能启动运行 | 是 / 否 / 未执行 |
| 阻塞问题数 | N |

## 自动化测试结论

| 项目 | 结论 |
|------|------|
| 状态 | completed / failed / blocked / skipped |
| 总用例数 | N |
| 通过 | N |
| 失败 | N |
| 跳过 | N |
| 通过率 | X% |

## 关键问题

| 问题 ID | 来源 | 严重度 | 类别 | 问题 | 证据 |
|---------|------|--------|------|------|------|
| BUILD-001 | build | 🔴 阻塞 | 📄 文档问题 | {问题} | [build-issues](build-verification/build-issues.md#build-001) |
| TEST-001 | auto-test | 🔴 阻塞 | 🔧 代码实现问题 | {问题} | [test-report](auto-test/test-report.md#test-001) |

## 问题详情

{由 `scripts/prepare-feishu-report-content.js` 将 `build-verification/build-issues.md` 合并到这里。}

## 自动化测试详情摘要

{摘取 auto-test/test-summary.md 和 test-report.md 的关键结论，不粘贴全部日志。}

## 附件

| 附件 | 说明 |
|------|------|
| 源码压缩包 | 清理依赖、构建产物、缓存和敏感信息后的 demo 源码 |

{插入压缩包}

{插入关键截图，或列出截图说明。}
```

## 校正后文档评测汇总报告

写入本地：

```text
doc-test-reports/{run-name}/doc-eval-corrected/doc-eval-summary.md
```

这是最终主报告。不要再生成或依赖 `接入测试综合报告`。读者应该能在本报告中一次看完核心结论、角色评分、全部问题摘要和全部问题详情。

写入飞书前，`scripts/prepare-feishu-report-content.js` 会将本地 H1 对齐为 manifest 中的飞书文档标题，例如：

```markdown
# 【文档评测报告】汇总报告
```

模板：

```markdown
# 【文档评测报告】汇总报告

## 总体结论

| 项目 | 结论 |
|------|------|
| 文档评测 | {加权平均分 X/5；N 个问题，M 个阻塞} |
| 构建与自动化测试 | {构建/测试结论和飞书链接；如无则写未执行/跳过} |
| 最终接入判断 | 可接入 / 有条件可接入 / 不可接入 |

一句话结论：{站在客户接入视角给出最终判断。}

## 角色评分（校正后）

{没有构建/自动化测试校正时，标题用 `## 角色评分`。}

| 角色 | 匹配度 | 加权平均分 | 维度评分 | 报告 |
|------|--------|------------|----------|------|
| 客户端开发 | ★★★★★ | X/5 | - 完整性：X/5<br>- 准确性：X/5<br>- 可执行性：X/5<br>- 排障支持：X/5 | {飞书链接或本地链接，写入前会替换为飞书链接} |

**请阅读各角色详细报告文档并评论问题解决方案！**

## 问题摘要

问题摘要必须合并所有来源的问题：角色评测、构建验证、自动化测试、构建/测试校正新增问题，以及需要说明的非文档归因问题。不要再拆出“集成验证新增文档问题”或“非文档问题说明”独立表格。

| ID | 严重程度 | 问题类型 | 问题 | 角色 | 来源 | 定位 |
|----|----------|----------|------|------|------|------|
| DOC-001 | P0 | 文档问题 | {问题摘要} | 客户端开发、技术支持 | 角色评测 + 构建验证 | {GitHub 链接} |

## 问题详情

问题详情必须与 `问题摘要` 顺序一一对应。每一项尽量照搬各角色报告中的问题详情描述，只做必要合并：增加角色字段、合并重复角色、补充构建/测试证据和校正结论。

### DOC-001 {问题标题}

- 严重程度：P0 / P1 / P2
- 问题类型：文档问题 / 链接问题 / 概念缺失 / 测试脚本问题 / 环境问题 / SDK 或测试产物问题 / Demo 实现问题
- 来源：角色评测 / 构建验证 / 自动化测试 / 构建测试校正
- 角色：客户端开发、技术支持
- 定位：{GitHub 链接；非文档归因问题可链接构建/测试报告、日志或相关产物}
- 构建/测试证据：{BUILD/TEST 链接；无则写未执行/未复现}

**原文内容**

{如角色报告中有原文摘录，照搬。}

**实际阅读结果**

{照搬或合并角色报告中的描述。}

**构建/测试校正结果**

{说明构建验证和自动化测试是否复现该问题、是否改变严重度、是否新增证据。}

**为什么是问题**

{照搬或合并角色报告中的说明；如构建/测试复现，补充真实阻塞点。}

**建议**

{照搬或合并角色报告中的建议；必要时补充具体修复位置、示例代码或告警说明。}
```

汇总报告不要包含以下章节：

- `集成验证校正记录`
- `集成验证校正`
- `集成验证新增文档问题`
- `非文档问题说明`
- `共性问题`
- `改进建议排序`
- `产物索引`
- `加权维度评分`
- `接入测试综合报告`

`角色评分` 表格的 `报告` 列在写飞书前必须使用飞书链接，不要保留本地相对路径。创建飞书骨架后，运行 `scripts/prepare-feishu-report-content.js` 从 `feishu-report-manifest.json` 读取各角色文档 URL 并替换。

## 校正后角色报告

写入本地：

```text
doc-test-reports/{run-name}/doc-eval-corrected/role-*.md
```

以原始 `doc-eval/role-*.md` 为底稿生成校正版角色报告。不要只在末尾追加校正结论；必须把构建验证和自动化测试结果融合回原报告正文，让读者在问题摘要和问题详情里直接看到校正后的判断。

校正方式：

1. 保留原角色报告的主体结构和原问题 ID。
2. `阶段结论` 中的综合评分、阻塞问题数、体验问题数、优化建议数和一句话结论，要按构建/测试证据重新计算或说明保持不变的原因。
3. 原始角色报告中已有的 `定位` GitHub 链接必须原样保留，不能改写成文件名、章节名、所有文档或跨角色引用。
4. `问题摘要` 表格增加一列 `构建测试校正后严重程度`。
5. `问题详情` 中保留原始严重度，同时增加构建/测试校正结论。
6. `为什么是问题` 和 `建议` 要根据构建/测试结果适当改写，不能只保留原始纸面评测判断。
7. 构建/测试证据证明不成立的问题，不删除，标记为降级或撤销，并解释原因。
8. 构建/测试新增且属于文档质量的问题，追加为新的 `DOC-N-xxx` 问题，并在关联问题中指向 BUILD/TEST 来源，同时补充新的 GitHub 定位。

校正版角色报告中的关键字段：

```markdown
## 问题摘要

| 问题 ID | 类别 | 原严重度 | 构建测试校正后严重程度 | 问题 | 建议 |
|---------|------|----------|--------------------------|------|------|
| DOC-001 | 📄 文档问题 | 🔴 阻塞 | 🟡 体验差 | {校正后的一句话问题} | {校正后的一句话建议} |

## 问题详情

### DOC-001: {问题标题}

- 定位：[docs/xxx.mdx:123](https://github.com/ZEGOCLOUD/docs_all/blob/main/docs/xxx.mdx?plain=1#L123)
- 章节路径：{可选}
- 类别：📄 文档问题 / 🔗 链接问题 / 📖 概念缺失
- 原严重度：🔴 阻塞 / 🟡 体验差 / 🟢 优化建议
- 构建测试校正后严重程度：🔴 阻塞 / 🟡 体验差 / 🟢 优化建议 / 已撤销
- 构建/测试证据：[BUILD-001](../build-verification/build-issues.md#build-001) / [TEST-001](../auto-test/test-report.md#test-001) / 未复现
- 关联问题：{关联的 BUILD/TEST/DOC 问题；无则写无}

**原文内容**

```markdown
{摘录文档原文，包含必要上下文。}
```

**实际阅读结果**

{保留或改写原角色视角下的阅读结果。}

**构建/测试校正结果**

{说明构建验证和自动化测试是否复现该问题、是否改变严重度、是否新增证据。}

**为什么是问题**

{结合角色阅读影响和构建/测试结果重新说明。比如：构建阶段已阻塞，所以保持🔴；自动化测试未阻塞但造成额外查找成本，所以降为🟡；证据表明原判断不成立，所以撤销。}

**建议**

{根据构建/测试后的真实影响给出建议。优先给出应补充到文档的内容、位置、示例代码或告警说明。}
```

不要在报告末尾保留 `## 集成验证校正记录`。构建/测试校正结果必须融合进 `问题摘要` 和 `问题详情` 正文。
