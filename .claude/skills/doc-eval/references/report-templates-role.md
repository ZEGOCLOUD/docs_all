# 角色评测报告输出格式

本模板用于 `doc-eval` 中单个角色的详细评测报告。

## 输出位置

角色报告写入：

```text
doc-test-reports/{run-name}/doc-eval/role-{role-id}.md
```

`run-name` 固定格式：

```text
{product}-{platform}-{scope}-{date}
```

示例：

```text
doc-test-reports/aiagent-web-quick-start-2026-06-05/doc-eval/role-client-dev.md
```

## GitHub 定位要求

所有文档问题的定位必须使用有效 GitHub 链接，不能只写本地路径或章节名。

要求：

1. 链接必须指向具体文件。
2. 链接必须尽量带行号，例如 `#L123`。
3. 链接文字建议写成 `path/to/file.mdx:123`。
4. 如果问题来自 MDX import 的源文件，链接必须指向实际源文件，而不是只指向壳文件。
5. 章节路径可以补充，但不能替代 GitHub 链接。

请参考 file-to-github-link skill 指引获取GitHub链接。

| 评测日期 | {date} |

| 字段 | 值 |
|------|----|
| 角色 | {角色名} |
| 目标 | {该角色阅读文档的目标} |
| 关注路径 | {该角色典型阅读路径} |
| 匹配度 | ⭐⭐⭐ / ⭐⭐ / ⭐ |

| 字段 | 值 |
|------|----|
| 状态 | completed / failed / blocked |
| 综合评分 | X/5 |

| 体验问题数 | N |
| 优化建议数 | N |
| 一句话结论 | {一句话概括该角色视角下的文档质量} |

| 信息架构 | X/5 | {一句话说明，引用具体章节或定位} |

## 角色阅读走查记录

本章节记录该角色按自己的阅读路径走查文档时的发现，不是固定流水线阶段。小节名称应来自角色定义中的关注路径，或本次评测实际采用的阅读步骤。

### {阅读步骤 1：如“确认前提条件”}

- 找到了：{具体内容}
- 没找到：{具体内容}
- 困惑点：{具体内容}

### {阅读步骤 2：如“查找初始化参数说明”}

- 找到了：{具体内容}
- 没找到：{具体内容}
- 困惑点：{具体内容}

## 集成验证输入（可选）

本章节只在该角色评测结果能为后续构建验证提供输入时填写。它不是要求 `doc-eval` 一定继续执行后续阶段。

### 建议构建验证关注点

1. {步骤 1}
2. {步骤 2}

### 可能阻塞构建验证的问题

|---------|----------|------|
| DOC-001 | {步骤名} | {为什么可能阻塞构建或测试} |

```

## 技术支持角色额外章节

技术支持角色除标准格式外，还需增加：

```markdown
## 模拟用户问题测试

| 用户问题 | 文档能否解答 | 定位 | 说明 |
|----------|--------------|------|------|
| {典型用户问题} | 是 / 否 | [docs/xxx.mdx:123](https://github.com/...) | {说明} |
```
