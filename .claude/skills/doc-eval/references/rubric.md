# 文档评测评分标准（Rubric）

## 开发者能力定位

**所有开发者角色（客户端开发、服务端开发、全栈开发）统一按"零经验开发者"定位。**
- 没有任何 ZEGO 产品经验（RTC、IM、AI Agent、数字人等均未使用过）
- 不熟悉实时通讯行业概念（如房间、用户 ID、Token、鉴权、回调等）
- 依赖文档解释一切术语和概念
- 不会猜测或假设行业惯例

## 文档类型定义

评测前必须先识别文档类型。根据目录名或文档内容判断：

### 匹配规则

1. **❌ 的角色不参与该文档评测**，无需启动对应 agent
2. **⭐⭐⭐ 的角色必须评测**，且评分权重使用完整权重
3. **⭐⭐ 的角色应评测**，评分权重正常
4. **⭐ 的角色可选评测**，权重降低（各维度取平均权重）

## 评测维度（6 个）

每个维度 1-5 分，必须基于以下描述打分，不可凭感觉。

### 1. 信息架构（章节逻辑、顺序、层次）

| 分数 | 描述 |
|------|------|
| 5 | 章节逻辑清晰，由浅入深，循序渐进，无冗余跳跃。读者可以线性阅读不迷失。 |
| 4 | 整体逻辑清晰，个别章节位置可以优化，但不影响理解。 |
| 3 | 大体通顺，部分章节顺序不合理（如高级主题出现在基础介绍之前），需要来回跳转。 |
| 2 | 结构松散，章节之间缺少逻辑关联，读者难以建立全局认知。 |
| 1 | 结构混乱，毫无章法，读者完全找不到想看的内容。 |

### 2. 内容完整性（该角色需要的信息是否齐全）

| 分数 | 描述 |
|------|------|
| 5 | 该角色需要的所有信息都有，包括边界情况、限制说明、常见问题等。 |
| 4 | 核心信息完整，部分次要信息缺失但不影响主流程。 |
| 3 | 有部分缺失，读者需要额外搜索或查阅其他文档才能完成目标。 |
| 2 | 关键信息缺失，读者无法仅靠本文档完成目标。 |
| 1 | 大量信息缺失，文档几乎没有参考价值。 |

### 3. 前提条件（前置要求是否明确）

| 分数 | 描述 |
|------|------|
| 5 | 明确列出所有前置要求（环境、版本、权限、前置知识），并说明如何满足。 |
| 4 | 列出了主要前提条件，个别细节缺失但可以推断。 |
| 3 | 有提到前提条件但不完整，读者可能在中途才发现缺少某些准备。 |
| 2 | 前提条件模糊或部分缺失，读者大概率会在中途卡住。 |
| 1 | 完全没有前提说明，读者无从准备。 |

### 4. 参数/配置说明（参数含义、默认值、示例是否充分）

| 分数 | 描述 |
|------|------|
| 5 | 每个参数都有类型、默认值、取值范围、说明和示例。配置项有完整说明。 |
| 4 | 大部分参数有说明，个别缺少默认值或示例。 |
| 3 | 有基本说明，但部分参数含义模糊或缺少必要上下文。 |
| 2 | 参数说明不全，读者需要猜测或查阅其他资料。 |
| 1 | 参数含义模糊或完全缺失。 |

### 5. 可发现性（能否快速找到所需内容）

| 分数 | 描述 |
|------|------|
| 5 | 通过目录、搜索、内部链接能一步定位到所需内容。相关主题之间有良好交叉引用。 |
| 4 | 能较快找到，偶尔需要多一次跳转。 |
| 3 | 能找到但需要多次跳转或搜索，缺少直接的交叉引用。 |
| 2 | 埋藏较深，需要耐心翻找，没有有效的导航辅助。 |
| 1 | 完全找不到，没有目录、索引或有效的搜索入口。 |

### 6. 准确性（信息是否与实际一致）

| 分数 | 描述 |
|------|------|
| 5 | 所有信息准确，与实际产品/SDK 行为一致，版本信息最新。 |
| 4 | 基本准确，有极个别措辞不精确但不影响使用。 |
| 3 | 有个别小错误或轻微过时，可能让读者产生困惑。 |
| 2 | 有明显错误或过时内容，可能误导读者。 |
| 1 | 严重错误，信息与实际严重不符。 |

## 评分校准规则

1. **起评分为 3** — 如果文档没有明显优点也没有明显缺点，默认 3 分
2. **必须有证据** — 每个评分必须引用文档中的具体内容作为依据
3. **问题必须定位** — 扣分项必须指出具体章节/段落位置
4. **亮点也要记录** — 做得好的地方也要记录，不仅仅是挑错
5. **按文档类型评判** — 评测时要考虑该文档类型的定位。例如：不应该在快速开始文档中期待完整的计费说明，那是计费文档的职责

## 术语检查维度

在"内容完整性"评分中，包含术语检查。**动态识别术语，不依赖硬编码列表。**

各角色评测时动态识别文档中的术语，检查文档是否解释：
1. 任何技术名词（如 SDK、API、Token、回调、鉴权）
2. 任何 ZEGO 特有概念（如房间、StreamID、用户 ID）
3. 任何行业特定术语（如推流、拉流、连麦）

| 评分 | 标准 |
|------|------|
| 5/5 | 所有首次出现的术语都有清晰解释 |
| 4/5 | 核心术语有解释，次要术语偶有遗漏 |
| 3/5 | 关键术语有解释，但部分术语未说明 |
| 2/5 | 只有少数术语有解释 |
| 1/5 | 几乎没有术语解释 |

**零经验开发者视角**：假设读者完全不了解 ZEGO 产品和实时通讯行业，文档必须解释一切。

**不要求解释的内容**：编程语言基础概念（变量、函数、类）、Web/移动端通用概念（HTTP、JSON、Promise）。

## 匹配度权重系数

汇总报告中，各角色评分按匹配度星星加权：

| 匹配度 | 系数 | 含义 |
|--------|------|------|
| ⭐⭐⭐ | 1.0 | 核心关注角色，评分全额计入 |
| ⭐⭐ | 0.6 | 重要但非核心，评分打 6 折 |
| ⭐ | 0.3 | 轻微关注，评分打 3 折 |
| ❌ | 0 | 不参与评测 |

### 汇总综合评分计算

```
综合评分 = Σ(各角色综合分 × 该角色匹配度系数) / Σ(匹配度系数)
```

示例：快速开始文档，3 个角色评测
- 客户端开发（⭐⭐⭐, 系数 1.0）综合分 3.8
- CTO（⭐⭐, 系数 0.6）综合分 4.0
- 产品经理（⭐, 系数 0.3）综合分 3.5

综合评分 = (3.8×1.0 + 4.0×0.6 + 3.5×0.3) / (1.0 + 0.6 + 0.3) = (3.8 + 2.4 + 1.05) / 1.9 ≈ 3.82
